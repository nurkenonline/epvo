## Методика выводов

- **Аналогия по имени:** `finishDate` в `Buildings` имеет описание → распространяем на `EducationBuildings` и `Dormitory`.
- **Аналогия по типу:** `ects` встречается в `Transcript` и `DeletedTranscript` с понятным описанием.
- **Контекст сущности:** поля `FOREIGN_SUBJECTS` связаны с академической мобильностью → `moneySource`/`moneySpent` = финансы, `profession`/`program` = специальность и программа обмена.
- **Тип данных:** строковые поля (`string`) в блоке `FOREIGN_SUBJECTS` указывают на свободный текст, а не FK-ссылки.

## Важные неочевидные правила валидации (ЕПВО)

По результатам анализа обращений в техподдержку ЕПВО (`chat_epvo.json`) выявлены критически важные скрытые правила валидации, из-за которых API отклоняет запросы:

### 1. `citizenshipId` (Гражданство)
- Поле **строго обязательно**, передача `null` недопустима.
- Отсутствие значения вызывает `500 Internal Server Error` или тихую пометку записи как ошибочной (студент не попадает в итоговые отчеты на финансирование).
- Если ИИН отсутствует (студент-иностранец), система идентифицирует его именно по паспорту и гражданству.

### 2. UPSERT и массовое сохранение (`/org-data/list/save`)
- Одиночный эндпоинт `/org-data/save` **отключен** (возвращает 404).
- Сохранение и обновление — это один и тот же вызов `/list/save` с передачей массива данных.
- ЕПВО автоматически определяет операцию сопоставляя **композитный ключ** сущности (например, `studentId`, `facultyId`). Если ключ найден в БД ЕПВО — обновляет все поля, если нет — создает.

### 3. Обязательная мультиязычность
- Для справочников типа `SUBJECT` (дисциплины), `FACULTIES`, `SPECIALIZATIONS` поля наименований и описаний (`nameRu`, `nameKz`, `nameEn`) **обязательны для всех трех языков**. Частая причина 500 ошибок при UPSERT.
