## 1. Видение

ОВПО обязаны передавать данные по ППС (профессорско-преподавательскому составу) в ЕПВО — это критично для отчётов **СУР**, **Национального рейтинга** и **Административных отчётов**. На данный момент в проекте есть каталог сущностей (RF_TFW-1.5) и краткая заметка о правиле гражданства (RF_TFW-4.A), но **отсутствует полная бизнес-логика маппинга** данных ППС, аналогичная тому, что сделано для студентов в TFW-4/TFW-5.

> Цель: Собрать полную картину — какие данные, в каких сущностях, с какой бизнес-логикой и в каком порядке ОВПО должно передавать в ЕПВО по преподавателям.

### 2.3. Что НЕ задокументировано (GAP)

1. **Скрытые поля** — `TUTOR_CAFEDRA` имеет поля, которых **нет в OpenAPI spec**, но которые критичны для отчётов:
   - `type` — тип занятости (0=штатный, 1=внутр. совместитель, 2=внешний совместитель)
   - `deleted` — признак увольнения
   - `primaryEmploymentID` — основное место работы (1=отечественный ОВПО, 2=зарубежный ОВПО)
   - `et_contract_start_date`, `et_contract_finish_date` — даты договора внешних совместителей

2. **Скрытые поля `TUTOR`** — не в OpenAPI spec:
   - `maternity_leave` — признак декретного отпуска
   - `ftutor` — признак «иностранный преподаватель»

3. **Бизнес-логика фильтрации ППС для отчётов СУР** — сложная multi-condition логика (OR из 4 блоков), подтверждённая Айдаром Буранбаевым в чате 30.04.2025.

4. **Порядок отправки данных** — аналог sequencing contract для студентов.

5. **Бизнес-правила `EMPLOYEE_ORDERS`** — какие типы приказов существуют (`CenterEmployeeOrderType` — приём, увольнение; неясно — декретные).

6. **Правила для `TUTOR_CAFEDRA_TRAINING_DIRECTIONS`** — один tutor_cafedra → несколько training directions (подтверждено в чате).

7. **Правила для `TUTOR_PUBLICATION`** — связь с `TUTOR` (через `tutorId`? через `puid`/`resourceId`?).

## 7. Принципы

1. **Источник = чат ЕПВО** — для скрытых полей (не в OpenAPI spec) единственный достоверный источник — чат техподдержки + инструкции МНВО.
2. **Полнота > детализация** — лучше задокументировать все поля с пометкой «требует верификации», чем пропустить.
3. **Формат RFC → TFW-4** — RF должен следовать формату, аналогичному RF_TFW-4.A (матрица маппинга + бизнес-правила).
