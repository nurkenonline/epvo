## 🟠 Вопросы для Фазы 2+ (НЕ блокируют Фазу 1)

### 1. Аутентификация в ЕПВО API
OpenAPI-спецификация **не описывает** механизм аутентификации. При попытке обращения к `/swagger-resources` получен **401 Unauthorized**.

**Вопросы:**
- Какой метод аутентификации: Bearer Token, OAuth2, API Key, сертификат?
Ответ: Basic Auth

- Есть ли у вас уже действующий токен?
Ответ: Нет

- Как получить токен? (Через личный кабинет ЕПВО? Через Platonus?)
Ответ: Через личный кабинет ЕПВО

- Токен постоянный или временный (с refresh)?
Ответ: Постоянный

### 2. universityId КазНПУ

## 🟠 Сомнения и риски

### 7. Batch size для /org-data/list/save
API принимает массив записей через `POST /org-data/list/save`. Неизвестно:
- Максимальный размер batch-запроса
- Есть ли rate limiting
- Как сервер обрабатывает частичные ошибки (атомарно ли сохранение?)

**Рекомендация:** Начать с batch size = 50, мониторить ответы, постепенно увеличивать.

**Ответ:** 
- Batch Size: 50 записей по умолчанию (настраивается в коде, Transcript = 1000).
- Rate Limiting: Отсутствует (синхронные запросы без задержек/retry).
- Атомарность: Частичная. EПВО возвращает failedRecords для ошибочных записей внутри батча. esuvoapi обрабатывает это, помечая ошибки локально (sendStatus = -1), успешные записи считаются сохранёнными.

### 8. UPSERT-поведение save
Документация подразумевает UPSERT (если запись есть — обновление, если нет — создание). Но:
- Не описано, по какому ключу определяется «уже существует»
- Не ясно, что происходит если отправить запись с тем же `typeCode` + PK, но другим `universityId`

**Рекомендация:** Проверить на небольших данных, прежде чем запускать масштабную загрузку.

**Ответ:** Вся логика дедупликации (UPSERT) лежит на стороне EПВО

### 9. Маппинг составных ключей
Правило из документации: «если НЕТ справочника [для type], используйте UNIVERSITY_ID_COMPOSITE_KEY» — неоднозначно.

**Рекомендация:** Для каждой сущности вручную проверить правильный composite key перед реализацией.

### 10. Идентификаторы (ID) и конфликты
Если у Univer и ЕПВО разные ID для одних и тех же сущностей (факультеты, кафедры), то:
- Нельзя просто копировать ID из Univer → нужен маппинг
- Или: ID в ЕПВО генерируются на стороне КазНПУ и являются уникальными в пределах `universityId`

**Вопрос:** Кто управляет ID? Вуз присваивает их сам или ЕПВО генерирует?

## 🟢 Рекомендации (что вы, возможно, не учли)

### 11. Системные справочники нужно загрузить ПЕРВЫМИ
Прежде чем отправлять данные, нужно скачать все системные справочники (`CommonDictionary`):
- Страны, национальности, языки, формы оплаты, типы вузов и т.д.
- Эти ID используются в полях OrgData-сущностей (напр. `sitizenshipId`, `sexId`, `studyLanguageId`)
- Без них невозможно правильно заполнить JSON-объекты

**Действие:** Первый шаг любой интеграции — `GET /common-dictionary/{code}/find-all` для всех ~50 справочников.

### 12. Трёхъязычность (RU/KZ/EN)
Большинство сущностей имеют поля `nameRu`, `nameKz`, `nameEn`. Если в источнике нет данных на одном из языков, нужна стратегия:
- Оставить пустое поле?
- Продублировать другой язык?
- Использовать машинный перевод?

### 13. Даты: формат yyyy-MM-dd
ЕПВО ожидает строго формат `"yyyy-MM-dd"` для дат и `"yyyy-MM-dd'T'HH:mm:ss"` для datetime. Если источник использует другой формат — нужна конвертация.

### 14. Поля с пометкой «Не указывать null»
Некоторые сущности (напр. `DeletedTranscript`, `Graduates`, `Student`) имеют обязательные поля с пометкой «не указывать null». Это значит:
- Поле должно быть в JSON
- Значение не может быть null
- Если данных нет в источнике — нужно решить, что ставить (0, пустая строка, значение по умолчанию)

### 15. Enrollment-эндпоинты (/ps/...) — другой функционал
Эндпоинты `/ps/magistracy-enrollment/findAll` и `/ps/grant-enrollment/findAll` — это **приём документов на поступление**, а не передача данных. Они работают в обратную сторону: вуз ЧИТАЕТ заявки абитуриентов. Не путать с основной задачей передачи OrgData.

### 16. Файлы и документы
Некоторые сущности (напр. `EmployeeOrders`) имеют поле `fileName`. Эндпоинт `/v1/file/{fileId}` позволяет скачать файл. Но:
- Формат загрузки файлов не описан в API
- Нужно ли загружать скан-копии приказов и других документов?

### 17. DeletedTranscript vs Transcript
В API есть `TRANSCRIPT` (для текущих обучающихся) и `DELETED_TRANSCRIPT` (для выпускников). Название «deleted» вводит в заблуждение — это не «удалённые», а «архивные» транскрипты выпускников. Обе сущности нужно передавать.
