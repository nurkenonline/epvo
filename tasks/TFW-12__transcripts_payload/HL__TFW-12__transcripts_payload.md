## 1. Vision
Определить точную структуру JSON payload для передачи академических транскриптов (оценок) в ЕПВО и спроектировать шаблон SQL-представления `v_epvo_transcripts` для локальной БД ИС ОВПО.

**Impact:** Интегратор (ВУЗ) получит готовый контракт данных, который останется только развернуть (адаптировать) в своей базе данных, что значительно упростит написание Python-кода синхронизации и предотвратит ошибки маппинга (Gap TD-1).

> "Теперь у нас есть единое плоское view, из которого скрипт просто забирает данные и отправляет в ЕПВО, без сложных join-ов на уровне Python."

## 2. Current State (As-Is)
- API ЕПВО требует передачи 30+ полей для каждой оценки в сущность `TRANSCRIPT` (а также `deletedtranscript` для выпускников).
- Локальная база данных ИС ОВПО хранит эти данные в сильно нормализованном виде (таблицы студентов, дисциплин, журналов, экзаменационных ведомостей).
- Согласно `RF_TFW-4.C`, формирование транскрипта на уровне кода (через ORM) приведет к серьезному снижению производительности и усложнению логики.
- Зафиксирован технический долг TD-1: до начала интеграции требуется создать `v_epvo_transcripts`.

## 3. Target State (To-Be)
- Чётко определенный JSON Payload для `TRANSCRIPT`, соответствующий документации ЕПВО (на основе `sur_doc.txt`).
- DDL-скрипт (черновик/спецификация) для SQL View `v_epvo_transcripts`.
- `v_epvo_transcripts` возвращает плоские записи, где каждая строка представляет одну итоговую оценку по дисциплине.

### 3.1 Result Visualization

> **To-Be: Поток данных**
```text
[ИС ОВПО (Журналы, Сессии)]
         |
         v
[SQL View: v_epvo_transcripts] --(Плоская таблица, 1 строка = 1 оценка)
         |
         v
[Python Sync Script] --(Простой UPSERT маппинг в dict)
         |
         v
[ЕПВО API: POST /org-data/list/save (typeCode=TRANSCRIPT)]
```

### 3.2 Value Flow

1. **INPUT**: Локальные таблицы БД (Оценки, Журналы, Расписание, Студенты).
2. **TRANSFORMATION**: Выполнение `v_epvo_transcripts` инкапсулирует всю бизнес-логику ИС ОВПО по определению итоговых оценок, пересдач и госэкзаменов.
3. **VALUE**: Готовый JSON Payload, валидный для ЕПВО (без 400 Bad Request из-за забытого обязательного поля `credits` или `subjectName`).

## 4. Phases

### Phase A: Definition & Payload Specification 🔴
- Анализ `sur_doc.txt` (Таблица 19 `transcript`) и OpenAPI.
- Составление эталонного JSON Payload.
- Проектирование контракта полей (колонок) для `v_epvo_transcripts`.

### Phase B: SQL View Template 🟡
- Написание чернового DDL-кода для создания `v_epvo_transcripts`.
- SQL будет предоставлен как шаблон, демонстрирующий, какие данные нужно объединить (с комментариями для адаптации DBA ВУЗа).

## 5. Definition of Done (DoD)
- ✅ 1. Спецификация JSON Payload для `TRANSCRIPT` полностью описана.
- ✅ 2. Спецификация контракта SQL-представления (выходные столбцы) подготовлена.
- ✅ 3. Написан базовый шаблон SQL-кода для `v_epvo_transcripts`.

## 6. Definition of Failure (DoF)
- ❌ 1. Payload не учитывает обязательные поля из `sur_doc.txt`.
- ❌ 2. Предлагаемое SQL View требует сложной дообработки (условной логики) на стороне Python.

**On failure:** Пересмотр структуры маппинга и возврат к изучению `sur_doc.txt`.

## 7. Principles
1. **Logic in DB, Not Code** — Вся бизнес-логика расчёта итоговых оценок, ECTS и типов дисциплин остается на стороне SQL-сервера ВУЗа.
2. **Flat is Better** — Python-скрипт должен только читать плоскую таблицу и сериализовать её в JSON, без обогащения данными "на лету".
3. **Vendor-Agnostic Structure** — SQL-шаблон должен оперировать исключительно абстрактными понятиями.
