# RF_TFW-4.D — Финансирование (Стипендия)

> **Дата**: 2026-03-11
> **Связанная задача**: TFW-4 Phase D
> **Статус**: 🟢 RF — Ожидает ревью координатора

## 1. Сводка
Данный документ описывает бизнес-логику для реализации интеграции модуля "Финансирование" (банковские реквизиты студентов и начисление стипендий) с ЕПВО (v3.0.1+). Описание основано на Постановлении Правительства РК (от 10.10.2022 № 799) и спецификации ЕПВО.

## 2. Банковские реквизиты (`Student` extension)
Передача банковских счетов необходима для выплаты стипендий и других траншей.

### 2.1 Бизнес-правило отправки
Банковские реквизиты (ИИК/БИК) **обязательны к передаче только для студентов, обучающихся по государственному образовательному заказу (на гранте)**.
Для студентов платного отделения (договорников) передача реквизитов не требуется (допускается отправка `null` или пропуск поля, если в локальной БД счета нет).

### 2.2 Маппинг полей (UPSERT `/org-data/list/save`)
При отправке профиля студента (`typeCode`: `"STUDENT"`) в блок массива добавляются следующие поля:

| EPVO Field | Type | Logic / Source | Валидация ЕПВО |
|------------|------|----------------|----------------|
| `iic` | Str | `local.student.iban` | **Строго:** 20 символов, начинается с `KZ`. Формат `/^KZ[0-9A-Z]{18}$/` |
| `bic` | Str | `local.student.bank_bic` | Строка, 8 символов (SWIFT BIC банка) |

*Fallback логика:* Если у грантника в локальной БД счет имеет неверный формат (например, 19 символов), скрипт интеграции должен **отфильтровать** этот счет (записать в лог ошибок) и отправить `iic: null`, чтобы избежать падения всего пакета `400 Bad Request`.

## 3. Назначение стипендий (`Scholarship`)
Сущность `Scholarship` отправляется ежемесячно для фиксации факта выплаты. ЕПВО строго валидирует право на стипендию на основе академических данных (из Фазы C).

### 3.1 Расчет `overall_performance` (Успеваемость)
Официальная шкала значений `overallPerformance` (из инструкции ЕПВО от 02.09.2024, строки 160–166):

| Значение | Условие |
|:--------:|---------|
| **5** | Все оценки за период — **отличные** |
| **4** | Есть хотя бы одна «хорошо», нет удовлетворительных/неудовлетворительных |
| **3** | Есть хотя бы одна удовлетворительная оценка, нет неудовлетворительных |
| **2** | Есть хотя бы одна неудовлетворительная оценка |
| **1** | Без учёта успеваемости (1 курс 1 семестр, докторантура, подгот. отделение) |
| **0** | Нет записей в транскрипте за период назначения |

> ⚠️ **Расхождение с RF:** Ранее в этом документе использовалась шкала 1–4 (из ПП РК №799). Официальная инструкция ЕПВО использует шкалу **0–5**. Приоритет — инструкция ЕПВО.

### 3.2 Привязка к приказу (Orders)
Согласно п.12 Правил, назначение стипендий производится **приказом руководителя организации**. ЕПВО требует передачи этого приказа (Order Type = 13).

**Бизнес-правило:** Ежемесячно ВУЗ издает локальный приказ на выплату. В ЕПВО отправляется `Scholarship`, которая жестко ссылается на `section_id` этого приказа.

### 3.3 Маппинг `Scholarship`
- **Endpoint**: `/org-data/list/save`
- **typeCode**: `"SCHOLARSHIP"`
- **Composite Key**: `UNIVERSITY_ID_COMPOSITE_KEY` → `{ type, id }`
- **Зависимость:** Сначала должен быть успешно сохранен `ORDER_SECTIONS`.
- **Гранулярность:** Одна запись = один месяц выплаты одному студенту.

| EPVO Field | Type | Обяз. | Logic / Source |
|------------|------|:-----:|----------------|
| `typeCode` | Str | ✅ | `"SCHOLARSHIP"` |
| `universityId` | Int | ✅ | ID вуза |
| `id` | Int | ✅ | Уникальный ID записи стипендии в локальной БД |
| `studentId` | Int | ✅ | `local.student.id` → FK на `STUDENT` |
| `scholarshipYear` | Int | ✅ | Год **выплаты** (например, 2026) |
| `scholarshipMonth` | Int | ✅ | Месяц **выплаты** (1–12) |
| `scholarshipPayDate` | Date | — | Дата фактической выплаты (`yyyy-MM-dd`) |
| `scholarshipMoney` | Num | ⚠️ | Сумма к выплате. **Де-факто обязательно** — без него 500 ошибка (см. KNOWLEDGE.md §2.9) |
| `scholarshipTypeId` | Int | ✅ | Тип стипендии из центрального справочника (значения могут меняться — проверять актуальность!). Из чата: ID=1 временно удалялся, затем вернулся (19.09.2023). |
| `sectionId` | Int | ✅ | `local.order_section.id` — ID секции приказа о **назначении** (→ `ORDER_SECTIONS` → `ORDERS`) |
| `additionalPayment` | Bool | — | Флаг доплаты (`true`/`false`) |
| `scholarshipAwardYear` | Int | ✅ | Учебный год, по итогам которого **назначена** стипендия (обязательно по инструкции от 02.09.2024) |
| `scholarshipAwardTerm` | Int | ✅ | Семестр, по итогам которого **назначена** (1 или 2) (обязательно по инструкции) |
| `overallPerformance` | Int | ✅ | Успеваемость: 5=отлично, 4=хорошо, 3=удовл., 2=неудовл., 1=без учёта, 0=нет данных (§3.1) |
| `terminationDate` | Date | — | Дата прекращения стипендии (при лишении) |
| `termination_ord_sectionid` | Int | — | ID секции приказа о **прекращении** стипендии |

**JSON-пример (из реальной интеграции, universityId=7):**
```json
{
  "typeCode": "SCHOLARSHIP",
  "universityId": 7,
  "id": 1,
  "studentId": 44386,
  "scholarshipYear": 2022,
  "scholarshipMonth": 1,
  "scholarshipPayDate": "2023-09-26",
  "scholarshipMoney": 41898.0,
  "scholarshipTypeId": 1,
  "terminationDate": null,
  "additionalPayment": false,
  "sectionId": 25515,
  "scholarshipAwardYear": 2022,
  "scholarshipAwardTerm": 2,
  "overallPerformance": 4,
  "termination_ord_sectionid": 0
}
```

> ⚠️ **Внимание:** Поле `scholarshipMoney` **не возвращается** при `find-all-pageable` (write-only), но без него сервер падает с 500. Подробности → `KNOWLEDGE.md` §2.9.

### 3.4 Предусловия для отображения в карточке финансирования
Чтобы приказ о назначении стипендии и данные по нему были видны в карточке финансирования, **все** нижеперечисленные условия должны быть выполнены:

1. **`STUDENT`**: `isinretire` = `0` или `1` (НЕ `null` — иначе студент не отобразится в финансировании вообще). Источник: чат ЕПВО, Бахтияр 23.09.2023.
2. **`STUDENT`**: `enrollOrderDate` — дата зачисления не должна быть из текущего семестра для не-первокурсников. Источник: чат ЕПВО, Евгений 21.09.2023.
3. **`STUDENT_INFO`**: `iic`, `bic` — заполнены для грантников.
4. **`ORDERS`** с `orderTypeId=13`: приказ о назначении стипендии загружен, `categoryId` заполнен.
5. **`ORDER_SECTIONS`**: приказ имеет хотя бы одну секцию.
6. **`ORDER_STUDENT_INFO`/`sectionpersons`**: студент привязан к секции, `movement_date` и `overall_performance` заполнены.
7. **`SCHOLARSHIP`**: записи созданы с корректным `sectionId` (ID **секции**, НЕ приказа), `scholarshipMoney`, `scholarshipTypeId`, `scholarshipAwardYear`, `scholarshipAwardTerm`, `overallPerformance`.
8. **Кэш ЕПВО**: после успешной отправки (200 OK) данные появляются с задержкой **~1–2 часа**.

### 3.5 Частые ошибки при интеграции стипендий (из чата ЕПВО)

| Проблема | Причина | Источник |
|----------|---------|----------|
| Приказ есть, но в финансировании пусто | `SCHOLARSHIP` не отправлен или `sectionId` = `orderId` (нужен ID секции) | Анализ payload |
| Вид стипендии не отображается | `scholarshipTypeId` невалидный (удалён из справочника) | чат, Даулет 19.09.2023 |
| Общая успеваемость пуста | `overallPerformance` не заполнен или неверное значение | чат, Евгений 19.09.2023 |
| Студент не попадает в отчёт | `isinretire = null` в `STUDENT` | чат, Бахтияр 23.09.2023 |
| Не-первокурсник не в отчёте | `enrollOrderDate` из текущего семестра | чат, Евгений 21.09.2023 |
| Опечатка в имени поля → данные не сохраняются | `shcolarshipAwardYear` вместо `scholarshipAwardYear` | чат, Евгений 19.09.2023 |
| Данные исчезли после обновления суммы | Full Replace: не все поля переданы при UPSERT | KNOWLEDGE.md §1.1 |
