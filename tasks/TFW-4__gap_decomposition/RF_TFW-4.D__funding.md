# RF_TFW-4.D — Финансирование (Стипендия)

> **Дата**: 2026-03-11
> **Связанная задача**: TFW-4 Phase D
> **Статус**: 🟢 RF — Ожидает ревью координатора

## 1. Сводка
Данный документ описывает бизнес-логику для реализации интеграции модуля "Финансирование" (банковские реквизиты студентов и начисление стипендий) с ЕПВО (v3.0.1+). Описание основано на Постановлении Правительства РК (от 10.10.2022 № 799) и спецификации ЕПВО.

## 2. Банковские реквизиты (`Student` extension)
Передача банковских счетов необходима для выплаты стипендий и других траншей.

### 2.1 Бизнес-правило отправки
Банковские реквизиты (ИИК/БИК) **обязательны к передаче только для студентов, обучающихся по государственному образовательному заказу (на гранте)**.
Для студентов платного отделения (договорников) передача реквизитов не требуется (допускается отправка `null` или пропуск поля, если в локальной БД счета нет).

### 2.2 Маппинг полей (UPSERT `/org-data/list/save`)
При отправке профиля студента (`typeCode`: `"STUDENT"`) в блок массива добавляются следующие поля:

| EPVO Field | Type | Logic / Source | Валидация ЕПВО |
|------------|------|----------------|----------------|
| `iic` | Str | `local.student.iban` | **Строго:** 20 символов, начинается с `KZ`. Формат `/^KZ[0-9A-Z]{18}$/` |
| `bic` | Str | `local.student.bank_bic` | Строка, 8 символов (SWIFT BIC банка) |

*Fallback логика:* Если у грантника в локальной БД счет имеет неверный формат (например, 19 символов), скрипт интеграции должен **отфильтровать** этот счет (записать в лог ошибок) и отправить `iic: null`, чтобы избежать падения всего пакета `400 Bad Request`.

## 3. Назначение стипендий (`Scholarship`)
Сущность `Scholarship` отправляется ежемесячно для фиксации факта выплаты. ЕПВО строго валидирует право на стипендию на основе академических данных (из Фазы C).

### 3.1 Расчет `overall_performance` (Успеваемость)
Согласно п.5 Главы 2 Правил (ПП РК № 799), стипендия назначается обучающимся, получившим по результатам сессии эквивалент оценок, соответствующий "отлично" и "хорошо" (A, A-, B+, B).

**Критическое правило (Фильтр оценок):**
Параметр `overall_performance` принимает значение `1` (соответствует требованиям), если в транскрипте студента за последнюю сессию **НЕТ** следующих буквенных оценок:
- `F` (Fail)
- `FX` (Fail with Retake)
- `D-`, `D`, `D+` (Удовлетворительно)
- `C-`, `C`, `C+` (Удовлетворительно)
- `B-` (Хорошо - *Внимание: базовая стипендия часто лишается при B-, но нужно сверять с локальным приказом, обычно B- это нижняя граница "хорошо", но на "повышенную" не проходит. В базовом маппинге ЕПВО отсутствие троек (C) и двоек (F) дает право на обычную стипендию.*)

**Псевдокод расчета:**
```python
def check_performance(student_transcripts):
    for mark in student_transcripts:
        if mark in ["F", "FX", "D-", "D", "D+", "C-", "C", "C+", "B-"]: # B- может быть плавающим в зависимости от типа стипендии
            return 0 # Не соответствует
    return 1 # Соответствует (только A, A-, B+, B)
```

### 3.2 Привязка к приказу (Orders)
Согласно п.12 Правил, назначение стипендий производится **приказом руководителя организации**. ЕПВО требует передачи этого приказа (Order Type = 13).

**Бизнес-правило:** Ежемесячно ВУЗ издает локальный приказ на выплату. В ЕПВО отправляется `Scholarship`, которая жестко ссылается на `section_id` этого приказа.

### 3.3 Маппинг `Scholarship`
- **Endpoint**: `/org-data/list/save`
- **Зависимость:** Сначала должен быть успешно сохранен `OrderSection`.

| EPVO Field | Type | Logic / Source |
|------------|------|----------------|
| `typeCode` | Str | `"SCHOLARSHIP"` |
| `id` | Int | Уникальный ID выплаты в локальной БД (или генерация `YYYYMM + student_id`) |
| `monthId` | Int | Месяц начисления (1-12) |
| `year` | Int | Год начисления (например, 2026) |
| `scholarshipSum` | Num | `local.scholarship.amount` (Сумма к выплате) |
| `orderSectionId` | Int | `local.order_section.id` (ID параграфа приказа) |
| `scholarshipTypeId`| Int | Тип (1 - Обычная, 2 - Повышенная, 3 - Президентская и т.д.) |
| `personId` | Int | `local.student.id` |

## 4. Архитектурные рекомендации (Observations & Risks)
| # | Type | Description |
|---|------|-------------|
| 1 | arch | **Батчинг (Batching):** Стипендии начисляются тысячам студентов одновременно. Скрипт интеграции должен формировать JSON массивы массивов (по 200-500 элементов `SCHOLARSHIP`) для предотвращения таймаутов на стороне серверов ЕПВО. |
| 2 | risk | **Отсутствие приказа:** Если локальная ИС ВУЗа физически не формирует электронный приказ (тип 13) каждый месяц, а начисляет деньги "скрытым" расчетом, интегратору придется синтезировать (генерировать) фиктивный "Приказ на стипендию" под капотом, иначе ЕПВО отклонит `Scholarship` из-за пустого `orderSectionId`. |
