# RF_TFW-4.C — Академические данные (Приказы и Оценки)

> **Дата**: 2026-03-07
> **Связанная задача**: TFW-4 Phase C
> **Статус**: 🟢 RF — Ожидает ревью координатора

## 1. Сводка
Данный документ описывает правила маппинга академических данных (Orders и Transcripts) из гипотетической нормализованной БД ВУЗа в REST API ЕПВО (v3.0.1+). Описание приведено в обобщенном виде, независимом от конкретной реализации (Platonus, Univer и др.).

## 2. Приказы (Orders - Композитная структура)
ЕПВО требует передачи приказов строго в виде реляционного дерева из 4-5 таблиц. ВУЗ подтвердил, что локальная БД хранит приказы в аналогичном иерархическом виде.

### 2.1 Иерархия отправки (Payload Structure)
Один логический приказ отправляется как массивы взаимосвязанных JSON объектов:
1. `Order` (Шапка приказа)
2. `OrderSection` (Параграфы/Секции приказа)
3. `SectionPerson` (Привязка студента к параграфу + Даты движения)
4. `OrderStudentInfo` (Академические параметры студента на момент приказа)
5. `OrdersAdditional` (Финансовые параметры, опционально)

### 2.2 Маппинг `Order` (Шапка)
- **Endpoint**: `/org-data/list/save`
| EPVO Field | Type | Logic / Source |
|------------|------|----------------|
| `typeCode` | Str | `"ORDERS"` |
| `id` | Int | Уникальный ID приказа (или сгенерированный) |
| `orderNumber` | Str | `local.order.number` |
| `orderType` | Int | Тип по справочнику ЕПВО (`local.order.type_id`). *Зачисление=1, Отчисление=3, Перевод=4* |
| `orderDate` | Date | `local.order.date` (ISO 8601) |

### 2.3 Маппинг `SectionPerson` (Движение + Мобильность)
| EPVO Field | Type | Logic / Source |
|------------|------|----------------|
| `typeCode` | Str | `"SECTION_PERSON"` |
| `sectionId` | Int | ID `OrderSection` |
| `personId` | Int | `local.student.id` |
| `movementDate` | Date | Дата вступления приказа в силу (`local.order.effective_date`) |
| `academicMobility`| Int | Флаг академ. мобильности. **Логика: `IF local.order.academic_mobility == true THEN 1 ELSE 0`** |

## 3. Оценки (Transcripts)
Учитывая сложность сбора оценок из разных локальных таблиц (сессии, журналы, дисциплины), интегратору ВУЗа **настоятельно рекомендуется** создать плоское SQL-представление `v_epvo_transcripts`.

### 3.1 Структура гипотетического View (`v_epvo_transcripts`)
Представление должно выдавать одну строку на каждую итоговую оценку по предмету за семестр:
* `student_id` (ID студента)
* `course_number` (Курс, на котором сдана дисциплина)
* `term` (Семестр. 1 - осенний, 2 - весенний, 3 - летний)
* `subject_name_ru/kz/en` (Наименование дисциплины)
* `subject_code` (Локальный код дисциплины)
* `credits` (Количество ECTS кредитов)
* `is_passed` (Сдан ли предмет успешно, boolean)
* `mark_letter` (Буквенная оценка A, B, C, D, F)

### 3.2 Правила для транскрипта
- **Endpoint**: `/org-data/list/save`
| EPVO Field | Type | Required | Logic / Source |
|------------|------|----------|----------------|
| `typeCode` | Str | ✅ | `"TRANSCRIPT"` |
| `accepted` | Int | ✅ | Всегда `1` (Оценка утверждена) |
| `courseNumber` | Int | ✅ | `v_epvo_transcripts.course_number` |
| `term` | Int | ✅ | `v_epvo_transcripts.term` |
| `credits` | Int | ✅ | `v_epvo_transcripts.credits` |
| `isPassed` | Bool | ✅ | `v_epvo_transcripts.is_passed` |
| `subjectNameRu`| Str | ✅ | `v_epvo_transcripts.subject_name_ru` |
| `...` | ... | ... | Остальные языки + `subjectCode` |

**Правило 1 (Проверка 100% заполненности):**
API ЕПВО возвращает ошибку 400, если хотя бы одно текстовое поле дисциплины (`subjectName*` или `code*`) равно `null`.
*Логика обертки (Fallback):*
```text
IF field IS NULL OR field == "":
    EPVO.field = "-"
```

**Правило 2 (Стипендиальная проверка):**
Оценки из транскрипта используются ЕПВО для проверки права на стипендию (`overall_performance`). Значение `overall_performance=1` проставляется сущности стипендии, только если в транскрипте за эту сессию нет оценок `F`, `D+`, `D-` (или согласно актуальным правилам МОН РК).

## 4. Observations & Risks

| # | Type | Description |
|---|------|-------------|
| 1 | risk | **Транзакционная целостность UPSERT:** При отправке композитного приказа (5 массивов), ошибка в одной строке `SectionPerson` может привести к тому, что ЕПВО сохранит шапку `Order`, но отклонит студента. Необходим механизм Retry и логирования `failedRecords` по каждому типу под-сущности в JSON-ответе. |
| 2 | risk | **Объем транскриптов:** На каждого студента приходится до 10-15 записей `Transcript` в семестр. Это означает миллионы записей в год. Интегратору следует настроить батчинг (отправка массивами по 100-500 записей за 1 HTTP POST запрос). |
