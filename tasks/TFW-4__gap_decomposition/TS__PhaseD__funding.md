## 1. Спецификация (Specification)
Этот документ задает технические рамки для Executor'а при написании артефакта `RF_TFW-4.D__funding.md`. RF должен служить строгим техническим заданием для программирования алгоритмов отправки данных о стипендиях и банковских счетах.

### 1.1. Ожидаемая структура RF
Итоговый RF документ должен содержать следующие разделы:
1. **Банковские реквизиты (`Student` extension):**
   - **Endpoint:** `/org-data/list/save`
   - Маппинг полей `iic` (IBAN) и `bic` (БИК банка).
   - Бизнес-правило: кому обязательно передавать счета (только грантникам?).
2. **Стипендии (`Scholarship`):**
   - **Endpoint:** `/org-data/list/save`
   - Матрица маппинга полей (год, месяц, сумма, тип стипендии).
   - Правило привязки стипендии к ID секции приказа (`section_id`).
3. **Бизнес-логика (Critical Rules):**
   - Псевдокод (IF/ELSE) для расчета `overall_performance` по оценкам (A-F) и проверки государственного обеспечения (`full_provision`).

### 1.2. Базовые правила маппинга для Phase D
Executor должен формализовать в RF:
- **Типы стипендий:** Как сопоставить локальные типы стипендий (Обычная, Повышенная, Президентская) со справочником ЕПВО.
- **Цикл отправки:** Нужно ли каждый месяц дублировать приказ, или стипендия отправляется привязкой к старому приказу? (Исторически скрипты ЕПВО требуют отдельной отправки сущности `Scholarship` ежемесячно).

## 2. Качественный контракт (Quality Contract)
| Правило | Обоснование |
|---------|-------------|
| Достоверность IBAN | Валидация ЕПВО отклоняет некорректные форматы IBAN (ошибка `TypeMismatch`). RF должен указать на необходимость регулярного выражения для проверки формата `KZ...` перед отправкой. |
| Изоляция логики успеваемости | Расчет стипендиальной метрики (`overall_performance`) должен быть строго описан математически/логически, чтобы программист мог реализовать его без погружения в академические правила. |

## 3. План верификации продукта (Verification Plan)
- **Review:** Координатор проверит логическую консистентность передачи стипендий с фазой C (где описано представление `v_epvo_transcripts`).
- Проверка наличия обработки кейса, когда у студента нет счета (допустимо ли отправлять `null`).
