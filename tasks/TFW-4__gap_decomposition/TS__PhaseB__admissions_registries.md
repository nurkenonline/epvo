## 1. Спецификация (Specification)
Этот документ описывает технические требования к разработке итогового мануала `RF_TFW-4.B__admissions_registries.md`.
Данный артефакт будет служить Source of Truth для программирования процессов синхронизации словарей и образовательных программ между ВУЗом и ЕПВО.

### 1.1. Ожидаемая структура RF
Итоговый RF документ должен содержать:
1. **Справочники (Dictionaries):**
   - Перечень статических справочников (Пол, Форма обучения, Язык и т.д.), которые мапятся "один-к-одному".
2. **EducationProgram (Образовательная программа):**
   - **Endpoint:** `/org-data/list/save`
   - Матрица маппинга полей.
3. **Бизнес-логика (Critical Rules):**
   - Условия ветвления (псевдокод) для сопоставления классификаторов специальностей (Например, старый классификатор ГОП vs новый).

### 1.2. Базовые правила маппинга для Phase B
Executor должен извлечь из GAP Analysis и формализовать в RF:
- **Сопоставление Profession (Специальности):** 
  - Как локальная база данных ВУЗа хранит специальность?
  - Как она сопоставляется с `center_profession` (например, по строковому коду `6B04101` или внутреннему ID)?
  - Что делать, если специальность не найдена в системной таблице ЕПВО? (Fallback логика или Drop).
- **Связь ОП и Квалификаций:** Описание того, как Educational Program привязывается к Degree/Qualification.

## 2. Качественный контракт (Quality Contract)
| Правило | Обоснование |
|---------|-------------|
| Псевдокод для маппинга | Избегать расплывчатых текстовых описаний. Интегратор должен понимать четкий алгоритм (JOIN или поиск по словарю в памяти). |
| Явное указание ключей | Для каждого словаря должно быть понятно, по какому полю идет стыковка (по `code`, по `id` или по `name`). |
