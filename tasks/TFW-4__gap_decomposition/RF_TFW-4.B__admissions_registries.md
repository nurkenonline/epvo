# RF_TFW-4.B — Реестр ОП и Справочники (Admissions and Registries)

> **Дата**: 2026-03-07
> **Связанная задача**: TFW-4 Phase B
> **Статус**: 🟢 RF — Ожидает ревью координатора

## 1. Сводка
Этот документ содержит бизнес-логику (Mapping Rules) для синхронизации статических справочников и образовательных программ (`EducationProgram`) ВУЗа с ЕПВО. Эти данные являются фундаментом: без корректной привязки к справочникам невозможно импортировать студентов и их оценки.

## 2. Статические Справочники (Hardcoded Dictionaries)
Для базовых словарей, которые редко меняются во времени, утверждена стратегия прямого маппинга (Direct/Hardcode mapping) на уровне кода интеграции.

### 2.1. Языки обучения (Languages)
По подтверждению от ВУЗа, локальная БД хранит прямую колонку с системными ID языков.
**Правило 1 (Прямой маппинг языков):**
```text
EPVO.languageId = local.language_id
```

### 2.2. Формы оплаты и обучения (Payment & Study Forms)
Если потребуется, данные справочники должны быть захардкожены в слое интеграции (например, Enum в Python/Java).
```text
// Пример Enum:
1 = Грант (Очная)
2 = Платное (Очная)
```

## 3. Сущность: EducationProgram / SPECIALIZATIONS (Образовательная программа)
*(В API ЕПВО фактический typeCode = "SPECIALIZATIONS". TD-25)*
- **Endpoint**: `/org-data/list/save` (или `/education-programs`)
- **Зависимости**: `CenterProfession` (Специальности / ГОП)

### 3.1. Матрица Маппинга
| EPVO Field | Type | Required | AD Source / Logic |
|------------|------|----------|-------------------|
| `code` | Str | ✅ | Уникальный код ОП внутри ВУЗа |
| `nameRu` | Str | ✅ | Название программы на русском |
| `nameKz` | Str | ✅ | Название программы на казахском |
| `duration` | Int | ✅ | Срок обучения в месяцах. См. Пр. 2. |
| `professionCode` | Str | ⚠️ | Код специальности. См. Пр. 3. |

### 3.2. Бизнес-правила (Critical Business Logic)

**Правило 2 (Вычисление срока обучения):**

> [!WARNING]
> **Семантический конфликт (TD-23)**:
> В `EducationProgram` (`SPECIALIZATIONS`) сроком обучения является поле `duration`, которое следует передавать в **месяцах**. В сущности `PROFESSION` (ГОП) есть поле `studyPeriod`, которое измеряется исключительно в **семестрах**. Будьте осторожны при маппинге.

В локальной БД срок обучения хранится в годах. API ЕПВО требует срок в месяцах.
```text
IF local.duration_years IS NOT NULL:
    EPVO.duration = int(local.duration_years) * 12
ELSE:
    LOG_WARNING("Срок обучения не указан для ОП: " + local.code)
    EPVO.duration = 48 // Fallback (стандартный бакалавриат 4 года)
```

**Правило 3 (Маппинг Специальностей / МСКО):**
По утверждению ВУЗа, прямая локальная колонка хранит код специальности, но он может не быть строго привязан к системным ID МСКО/ОКЭД (классификаторам ЕПВО). Принято решение отправлять данные "как есть".
```text
EPVO.professionCode = local.profession_code 
// Внимание: Если ЕПВО производит строгую сверку professionCode на своей стороне 
// со справочником center_profession, возможны ошибки 400 Bad Request.
```

## 4. Observations & Risks

| # | Type | Description |
|---|------|-------------|
| 1 | risk | **Старые коды специальностей (Historical Data)**: Вопрос интегратора был вызван спецификой РК. До 2018-2019 гг. специальности кодировались по-старому (например `5B042000`). Сейчас используется новый классификатор ГОП (например `6B04101`). Если попытаться отправить в ЕПВО старый код (`5B...`), который отсутствует в их современном справочнике `center_profession`, ЕПВО может вернуть ошибку валидации. Интегратор должен быть готов к этому edge-case при интеграции исторических данных выпускников. |
