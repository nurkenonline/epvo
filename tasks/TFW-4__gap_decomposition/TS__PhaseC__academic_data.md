## 1. Спецификация (Specification)
Этот документ задает технические рамки для Executor'а при написании артефакта `RF_TFW-4.C__academic_data.md`. RF должен служить строгим техническим заданием для программирования алгоритмов отправки приказов и сессионных оценок.

### 1.1. Ожидаемая структура RF
Итоговый RF документ должен содержать следующие разделы:
1. **Транскрипты (Transcripts / Grades):**
   - **Endpoint:** `/org-data/list/save` (или `/transcripts...`)
   - Матрица маппинга полей.
   - Правила расчета/фильтрации оценок, включая логику `overall_performance` (буквенные оценки).
2. **Приказы (Orders & OrderStudentInfo):**
   - **Endpoint:** `/org-data/list/save`
   - Матрица маппинга.
   - Таблица маршрутизации (соответствия) локальных статусов студентов системным кодам типов приказов ЕПВО (Зачисление, Отчисление и т.д.).
3. **Бизнес-логика (Critical Rules):**
   - Псевдокод (IF/ELSE) для определения дат начала/конца действия приказа.
   - Механика избежания дубликатов при отправке транскриптов (уникальные композитные ключи).

### 1.2. Базовые правила маппинга для Phase C
Executor должен формализовать в RF следующие аспекты:
- **Оценки:** Как фильтруются оценки, подлежащие отправке? Нужно ли передавать "зачеты" или только экзаменационные оценки (A-F)? Как локальный ВУЗ хранит пересдачи?
- **Академическая мобильность:** Логика заполнения поля `academic_mobility` при приказах о направлении в другой ВУЗ.
- **Синхронизация:** Учитывая объем оценок, какая стратегия рекомендуется (массовый UPSERT или дифференциальная загрузка)?

## 2. Качественный контракт (Quality Contract)
| Правило | Обоснование |
|---------|-------------|
| Строгая валидация | Оценки и приказы - самая частая причина 400 ошибок API ЕПВО (например, ConstraintViolation). RF должен перечислить *все* `required` поля, без которых отправлять запись запрещено. |
| Изоляция логики | Логика определения типа приказа не должна зависеть от сырого названия приказа в БД, а опираться на нормализованные ID или справочники. |
