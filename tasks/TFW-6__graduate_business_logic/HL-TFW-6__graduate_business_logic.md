# HL — TFW-6: Бизнес-логика перевода обучающегося в статус «Выпускник»

> **Дата**: 2026-03-15

## 1. Видение

Описать полную бизнес-логику перевода обучающегося в статус «Выпускник» в зависимости от уровня обучения (`degreeId`), включая: выбор типа приказа, pipeline сущностей, заполнение дипломных данных в трёх точках ввода, и формирование профиля `GRADUATES`.

> Цель — единый справочный документ, который интегратор может использовать как чеклист при реализации выпускной логики для бакалавриата, магистратуры и докторантуры.

## 2. Текущее состояние (As-Is)

Информация о выпускниках рассредоточена по нескольким документам:

**Ключевые проблемы:**
1. **Нет единого документа**, описывающего какой тип приказа использовать для каждого уровня обучения
2. В `RF_TFW-4.E` §6.1 фигурирует **ошибочный orderType=6** (зафиксировано в `RF_TFW-5` Observation #4, но не исправлено) — для бакалавриата корректно type=3 cat=1, для магистратуры/докторантуры — type=28
3. **Условные поля GRADUATES** по degreeId описаны в `RF_TFW-4.E` правилами 7-8, но не организованы как единая матрица

## 3. Целевое состояние (To-Be)

Единый RF-документ, содержащий:

### 3.1 Scope (уровни обучения)

| degreeId | Уровень | В scope |
|:--------:|---------|:-------:|
| 1 | Бакалавриат | ✅ |
| 2 | Магистратура (научно-педагогическое направление) | ✅ |
| 3 | Магистратура (профильное) | ✅ |
| 6 | Докторантура PhD | ✅ |
| 7 | Докторантура (профильное направление) | ✅ |
| 4 | Интернатура | ❌ Вне scope |
| 5 | Резидентура | ❌ Вне scope |
| 0 | Подготовительное отделение | ❌ Не выпуск |
| 10 | Подготовительное отделение (альт.) | ❌ Не выпуск |

### 3.2 Матрица «Уровень обучения → Тип приказа»

| degreeId | Уровень | orderType | categoryId | Примечание |
|:--------:|---------|:---------:|:----------:|------------|
| 1 | Бакалавриат | **3** | **1** | Отчисление по выпуску (спец. случай) |
| 2 | Магистратура (НПН) | **28** | **160** | Завершение обучения |
| 3 | Магистратура (профил.) | **28** | **160** | Завершение обучения |
| 6 | Докторантура PhD | **28** | **161** | Завершение обучения |
| 7 | Докторантура (профил.) | **28** | **161** | Завершение обучения |

> ⚠️ **Разрешение противоречия:** В `RF_TFW-4.E` §6.1 указан `orderType=6` — это ошибка. `orderType=6` = академический отпуск. Корректные значения: бакалавриат → type=3 cat=1; магистратура → type=28 cat=160; докторантура → type=28 cat=161. Источник: `RF_TFW-5` §3.10, `toepvo_dict_ORDER_CATEGORY.json`.

### 3.3 Полный pipeline (5 шагов — единый для всех уровней)

```
Шаг 1: TRANSCRIPT      (все оценки за финальную сессию → используются для GPA)
         ↓
Шаг 2: ORDERS           (orderTypeId = 3 или 28, см. матрицу)
      + ORDER_SECTIONS   (orderId, categoryId)
      + SECTION_PERSON   (sectionId, personId=studentId, movementDate ✅)
      + ORDER_STUDENT_INFO (sectionId, studentId, degreeId)
         ↓
Шаг 3: STUDENT update   (status=4, courseNumber=0)
         ↓
Шаг 4: STUDENT_DIPLOMA_INFO (diplomaSeries, diplomaNumber, diplomaDate)
         ↓
Шаг 5: GRADUATES        (полный профиль, ~20 not-null полей)
```

**Обоснование порядка:** Шаг 1 → GPA вычисляется из транскриптов. Шаг 2 → приказ регистрируется до смены статуса (иначе orphan). Шаг 5 последним → ссылается на studentId и ожидает status=4.

### 3.4 Условные поля GRADUATES по degreeId

| degreeId | Дополнительные обязательные поля |
|:--------:|----------------------------------|
| 1, 2, 3 | Стандартный набор (~20 not-null полей) |
| 6, 7 (докторантура) | + `doctorDefended`, `defenseOfDissertationInOtherHpeo`, `dateDissertationDefense`, `defenseOfDissertationOtherHpeoId` |

### 3.5 Три точки ввода дипломных данных

| Сущность | Поля | Обязательность | Назначение |
|----------|------|:--------------:|------------|
| `GRADUATES` | `iacDiplomaSeries`, `iacDiplomaNumber`, `diplomaDate` | ✅ not null | Профиль выпускника |
| `STUDENT_DIPLOMA_INFO` | `diplomaSeries`, `diplomaNumber`, `diplomaDate` | — | Карточка студента |
| `STUDENT_INFO` | `diplomaSeries`, `diplomaNumber`, `diplomaDate` | — | Общие доп.данные |

> ⚠️ Данные во всех трёх сущностях **ДОЛЖНЫ** совпадать. Рассогласование вызовет ошибку при проверке на `https://epvo.kz/#/verification_education_document`. Источник данных должен быть **единым** (TD-PHASE-E-2 из REVIEW PhaseE).


### 3.6 Ключевые бизнес-правила (из RF_TFW-4.E)

1. **GPA** — обязателен для `enrollYear >= 2010`
2. **ИИН** — обязателен для граждан РК (citizenshipId=113) и лиц с видом на жительство
3. **ГОП vs CenterProfessionCode** — взаимоисключающие (правило `centerProfChecked`)
4. **Фильтрация** — выпускник → GRADUATES только если `status=4`, `courseNumber=0`, `degreeId ∉ {0,10}`, диплом заполнен
5. **Серия/номер диплома** — если пустые, запись не отправляется (DROP_RECORD)
6. **ОП (specializationId)** — обязателен, если profession.classifier=2 (ГОП)

1. **Единый источник правды** — RF должен быть самодостаточным, не требовать перехода в другие документы
2. **Разрешение противоречий** — все расхождения между источниками явно разрешены с указанием, какое значение верно и почему
