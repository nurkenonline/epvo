## 1. Цель
Описать первый транш API-эндпоинтов ЕПВО (OrgData и CommonDictionary) согласно OpenAPI-спецификации, а также зафиксировать обнаруженные расхождения (Gap Analysis) между сухой спецификацией и детальной бизнес-логикой из полученных исторических инструкций (СУР, Административные отчеты, Финансирование, ОСМС).

## 2. Scope

### In Scope
- Документирование эндпоинта аутентификации (формат Basic Auth, передача `universityId`).
- Документирование эндпоинтов OrgData (`/org-data/find-all-pageable`, `/org-data/find-by-id`).
- Документирование эндпоинта CommonDictionary (`/common-dictionary/find-all`).
- Формирование теоретических примеров запросов (Request) и ответов (Response) на основе OpenAPI 3.0.1.
- **Mapping Gap Analysis**: Создание раздела, фиксирующего скрытую бизнес-логику из инструкций (СУР, АдмОтчеты, Финансы), которая не описана в OpenAPI. Включая (но не ограничиваясь): дополнительные поля статусов, условия обязательности, логику расчета успеваемости для стипендий, расширенные поля контингента и справочников.

### Out of Scope
- Документирование эндпоинтов сохранения (`/save`, UPSERT) и удаления (`/delete`) — это Phase B.
- Документирование эндпоинтов Enrollment (магистратура, гранты) и File API — это Phase C.
- Написание реального кода (скриптов) для проверки API. Только Markdown-теория.

## 4. Детальные шаги

### Step 1: Проектирование RF_TFW-2.1__orgdata_api.md
Создать файл и описать:
* Общие заголовки и Auth.
* `GET /org-data/find-all-pageable` (Query параметры: category, page, size, sort. Формат ответа Pageable).
* `GET /org-data/find-by-id` (Path параметр: id. Формат ответа Entity).
Теоретические примеры JSON составляются на основе схем из TFW-1.

### Step 2: Проектирование RF_TFW-2.2__common_dictionary_api.md
Создать файл и описать:
* `GET /common-dictionary/find-all` (Query параметр: category. Формат ответа).
Примеры JSON.

### Step 3: Проектирование RF_TFW-2.3__gap_analysis.md
Создать файл, где свести воедино "скрытые" требования из 4-х файлов инструкций:
* Раздел 1. Дополнительные поля контингента (students, student_info), отсутствующие в базовой спецификации (например, `incorrectiin`, `conditionally_enrolled`, `winter_admission`).
* Раздел 2. Специфичная логика модулей (например, `overall_performance` из Финансирования).
* Раздел 3. Условная обязательность полей в зависимости от `degreeid`, `examType` и т.д.

## 6. Риски фазы

| Риск | Mitigation |
|------|------------|
| Примеры JSON могут быть неполными из-за отсутствия живого сервера | Использовать максимально полную генерацию на основе OpenAPI-схем, явно указывая, что это теоретический пример. |
| Слишком большой объем Gap Analysis (4 инструкции) | Сфокусироваться только на самых важных/критичных расхождениях, влияющих на структуру JSON-сообщения (доп. поля, обязательность), не погружаясь в детальную текстовую шелуху отчетов. |
