## 1. Что сделано
Данный документ фиксирует расхождения (Gap Analysis) между:
А) Строгой `OpenAPI 3.0.1` спецификацией ЕПВО (2022 года).
Б) Реальными бизнес-требованиями из 4-х файлов инструкций интеграции ОВПО:
- `Инструкция_Показатели_СУР...pdf` (СУР)
- `Инструкция_Адм_отчеты...pdf` (АдмОтчеты)
- `ОВПО_Инструкция_для_передачи_данных_модуль_Финансирование...txt` (Финансы)
- `Инструкция для ОСМС...pdf` (ОСМС)

## 2. Неучтенные сущности и таблицы (Расширение схемы данных)
В OpenAPI описаны базовые `OrgData`, `CommonDictionary` и `Student`. Однако инструкции (СУР, АдмОтчеты) требуют маппинга более **60 таблиц**.
Ключевые сущности, передаваемые в ЕПВО (формируется как JSON Payload для скрытых/недокументированных эндпоинтов ЕПВО), которых нет в публичной спеке:
* `students` (Контингент, базовая таблица)
* `student_info` (Детальная информация о студенте, сертификаты, нострификация)
* `professions` (Специальности/ГОП ВУЗа)
* `studyforms` (Формы обучения ВУЗа)
* `orders` (Выпущенные приказы)
* `dormitories` (Общежития)
* `tutors` (Преподаватели) и `tutorqual` (Повышение квалификации ППС)
* `graduates` (Выпускники) и др.

## 3. Расширение полей контингента (`students` и `student_info`)
При отправке данных студента в ЕПВО, помимо стандартных полей (ФИО, ИИН, Пол), инструкции требуют десятки дополнительных полей бизнес-логики.

### 3.1 Модуль Финансирование (Стипендии)
Поля, критически важные для начисления гос. выплат:
* `students.iic` (ИИК/IBAN карточного счета)
* `students.bic` (БИК банка)
* `overall_performance` (Средний балл / Успеваемость) — логика расчета: сумма оценок (буквенных баллов от A до F) по всем предметам сессии. Для получения стипендии не должно быть оценок ниже "B-", "C+", "C", "C-", "D+", "D-", "F".
* `studentbenefits.full_provision` (Полное гос. обеспечение).
* `winter_admission` (Зимний набор).

### 3.2 Модули СУР и АдмОтчеты (Валидация и Статусы)
Поля с жесткими условными правилами (Conditional validation):
* `academic_mobility` — статус по мобильности.
* КАТО коды (`birth_place_cato_id`, `living_place_cato_id`, `registration_place_cato_id`) — обязательны только если гражданство = РК (`sitizenshipid = 1`).
* `incorrectiin` — логический флаг некорректного ИИН (true/false).
* `contractid` — источник финансирования (заполняется только при коммерческом обучении за счет работодателя).
* `sum_points`, `sum_points_creative` — баллы ЕНТ (обязательны только если `student_info.examtype = ЕНТ`).
* `conditionally_enrolled` — условно зачисленные (true, если баллы ЕНТ ниже проходного: `students_admission_subjects.points < minPoints`).

### 3.3 Модуль ОСМС
Для модуля ОСМС передается статус студента (`students.isstudent`):
* `1` - Обучающийся
* `2` - Абитуриент
* `3` - Выпускник/Отчислен (Если `coursenumber=0` = Выпускник, иначе Отчислен).

## 4. Ключевые выводы (Risks & Decisions)
1. **Проблема OpenAPI**: Спецификация OpenAPI ЕПВО описывает лишь CRUD справочников. Она абсолютно **не отражает** реальную транзакционную структуру данных (json payloads для приказов, контингента, стипендий, счетов), требуемую от ОВПО.
2. **Сложная валидация**: Маппинг данных из локальной ИС ВУЗа (Универ/Платонус) в ЕПВО требует сложных SQL-джойнов и IF/ELSE логики (например: передавать поле "Номер сертификата ЕНТ" только если студент бакалавр И тип поступления = ЕНТ). Нельзя просто "скопировать" таблицу в таблицу.
3. **Решение для архитектуры**: Вся эта "скрытая" логика валидации должна быть инкапсулирована в слое **Сверки (Reconciliation) и трансформации (dbt / Python)** внутри ClickHouse ДО отправки в ЕПВО, так как прямое перетекание данных из сырых таблиц ВУЗа невозможно.
