## 1. Видение
Описать каждый API-эндпоинт ЕПВО (OrgData, CommonDictionary, Enrollment) теоретически, опираясь строго на OpenAPI-спецификацию и документацию Platonus.
> "Чистая теория и полная ясность форматов без необходимости прямого подключения."

## 2. Текущее состояние (As-Is)
Схемы данных OrgData описаны в задаче TFW-1. Однако сами методы API (URL, параметры запросов, заголовки, структуры ответов, правила пагинации) разрозненны и находятся либо в виде сырого OpenAPI JSON, либо в `TASK.md` в виде краткой таблицы. 
Единого справочника по *поведению* эндпоинтов нет.

Кроме того, обнаружена документация (`Инструкция для ОСМС`), описывающая практический маппинг полей из любой информационной системы ОВПО в API ЕПВО. Эта инструкция содержит бизнес-логику (например, вычисление статуса `isinretire` или поля `winter_admission`, разделение серий дипломов интернатуры), которая **не отражена напрямую** в OpenAPI спецификации ЕПВО.

## 3. Целевое состояние (To-Be)
Подробная документация (справочник эндпоинтов) в формате Markdown, описывающая:
- **OrgData API**: `/find-all-pageable`, `/find-by-id`, `/list/save`, `/delete`.
- **CommonDictionary API**: `/find-all`.
- **Enrollment / PS API**: `/magistracy-enrollment/findAll`, `/grant-enrollment/findAll`.
- Примеры JSON-запросов и ответов.
- **Сводка расхождений (Mapping Gap Analysis)**: Фиксация разницы между строгой OpenAPI-спецификацией ЕПВО и скрытой бизнес-логикой из найденных инструкций (СУР, Административные отчеты, ОСМС, Финансирование и др.), включая расчёт статусов, льгот, успеваемости и расширения таблиц.
- Теоретическое описание обработки ошибок.

## 4. Фазы

Для соблюдения Scope Budget задача разбивается на фазы по группам эндпоинтов.

### Phase A: OrgData, CommonDictionary & Gap Analysis 🔴
- Документирование эндпоинтов чтения (`find-all-pageable`, `find-by-id`, `find-all`).
- Сравнение полей из массивных инструкций (СУР, Административные отчеты, Финансирование, ОСМС) с OpenAPI-схемой ЕПВО (выявление десятков неявных полей: КАТО, статусы прибытия, `winter_admission` и детализированные таблицы `student_info`, `orders` и расширения контингента).
- Документирование заголовков аутентификации (Basic Auth) и параметра `universityId`.
- Описание структуры ответов (включая поля пагинации).

### Phase B: Запись (Save) и Удаление (Delete) 🟡
- Документирование массового сохранения (`/org-data/list/save`).
- Описание структуры `failedRecords` и теоретического механизма UPSERT.
- Документирование эндпоинта `/delete`.

### Phase C: Специализированные эндпоинты (Enrollment / File) 🟢
- Документирование приёмных кампаний (бакалавриат, магистратура).
- Документирование эндпоинта работы с файлами (`/v1/file/{fileId}`).

## 5. Definition of Done (DoD)
- ✅ 1. Создан RF-файл (или несколько) с полным описанием всех методов API.
- ✅ 2. Для каждого метода представлены теоретические примеры `Request` и `Response`.
- ✅ 3. Документированы механизмы пагинации (теоретически, на основе OpenAPI).
- ✅ 4. Документированы механизмы аутентификации (теоретически, Basic Auth).

## 6. Definition of Failure (DoF)
- ❌ 1. OpenAPI-спецификация недоступна или не содержит достаточной информации для формирования примеров.

**При провале:** Использовать доступную документацию Platonus (PDF/Word), если она есть в контексте проекта, либо запросить уточнения у Координатора (человека).

## 7. Принципы
1. **No Code** — Никаких скриптов, только Markdown-документация.
2. **Clear Examples** — Каждый эндпоинт обязан сопровождаться блоком кода с примером JSON.
3. **Traceability** — Любые предположения (если спецификация неясна) должны помечаться тегом `> ⚠️ Предположение`.
