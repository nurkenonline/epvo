## 1. Understanding (как понял задачу)
Задача Фазы А — задокументировать эндпоинты чтения справочников (`OrgData`, `CommonDictionary`) на основе OpenAPI-спецификации ЕПВО (извлеченной в TFW-1), а также провести **детальный Mapping Gap Analysis**. Gap Analysis должен выявить и зафиксировать расхождения между сухой спецификацией API и реальными требованиями к передаваемым данным, описанными в четырех массивных исторических инструкциях (СУР, Административные отчеты, Финансирование, ОСМС). Результатом будут 3 MD-файла с теоретической документацией и JSON-примерами.

## 2. Entry Points (откуда начинать)
- OpenAPI схемы (`schemas.json`, извлеченные в первой задаче).
- Текстовые выжимки из инструкций (`doc_pdf.txt`, `sur_doc.txt`, `adm_doc.txt`, `ОВПО_Инструкция...txt`).
- Создание файлов в папке `tasks/TFW-2__api_endpoints/`.

## 4. Recommendations (suggestions, not blocking)
1. **Объем Gap Analysis**: Из-за огромного размера инструкций (в сумме более 20 тыс. строк) рекомендуется в документе Gap Analysis (RF_TFW-2.3) приводить не каждую таблицу целиком, а именно *разницу* (добавленные поля, статусы) и примеры комплексной логики, чтобы не превращать документ в нечитаемую простыню 1 в 1 копирующую исходники.

## 5. Risks Found (edge cases, potential issues not in TS)
1. Отсутствие реального тестового контура ЕПВО делает все JSON-примеры строго теоретическими (mock). Возможны мелкие расхождения форматов (например, форматы дат) в реальной интеграции.

## 6. Inconsistencies with Code (spec vs reality)
- OpenAPI спецификация на данный момент вообще не отражает реальную сложность и многообразие систем, интеграция с которыми описана в документах СУР/Адм. Это главное несоответствие, которое ляжет в основу Gap Analysis.
