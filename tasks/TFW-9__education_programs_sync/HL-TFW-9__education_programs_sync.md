# HL — TFW-9: Инструкция по синхронизации Реестра образовательных программ

> **Дата**: 2026-04-15
> **Автор**: Coordinator (AI)
> **Статус**: ✅ HL — Утверждён

---

## 1. Видение
Интеграторам ИС ОВПО нужна исчерпывающая инструкция по передаче данных «Реестра образовательных программ» в ЕПВО. Текущая документация (RF_TFW-4.B) покрывает только 5 полей одной сущности. Фактически реестр ОП состоит из **7 взаимозависимых сущностей** (RF_TFW-1.2), и без полного описания sequencing, composite keys, lookup-ов системных справочников и обработки edge-cases интегратор не сможет корректно загрузить данные.

> Реестр ОП — фундамент: без него невозможно создать студентов, приказы и выпускников.

## 2. Текущее состояние (As-Is)

| Артефакт | Что покрывает | Проблема |
|----------|---------------|----------|
| RF_TFW-1.2 (Entity Catalog) | 7 сущностей: поля, типы, JSON-примеры | Каталог, а не инструкция. Нет бизнес-логики, нет sequencing, нет gotchas |
| RF_TFW-4.B (Gap Analysis Phase B) | 5 полей `EducationProgram`, 2 правила | Слишком краткий. Нет `PROFESSION`, нет `centerProfId` lookup, нет обработки ошибок |
| RF_TFW-1.14 (System Dictionaries) | `CenterProfession` упомянут в перечне | Нет описания структуры справочника, нет примера lookup |
| KNOWLEDGE.md | Нет раздела про ОП/ГОП | Gap — реестр программ не документирован |

**Итого:** Интегратор получает каталог полей, но не знает:
- В каком порядке создавать сущности
- Как найти правильный `centerProfId` из системного справочника
- Что делать с кодами специальностей до 2018 года
- Какие поля реально обязательны (не null-safe), а какие – optional

## 3. Целевое состояние (To-Be)

Один самодостаточный RF-документ (`RF__TFW-9__education_programs_sync.md`), который содержит:

1. **Полный маппинг** всех полей для каждой из ключевых сущностей (`PROFESSION`, `SPECIALIZATIONS`, `PROFESSION_CAFEDRA`, `PROFESSION_COST`, `TYP_CURRICULUM`) с бизнес-правилами заполнения (`IF/ELSE`)
2. **Sequencing Contract** — строгий порядок UPSERT сущностей (с учётом FK-зависимостей)
3. **Lookup-инструкция для `CenterProfession`** — как получить `centerProfId` через системный справочник, алгоритм сопоставления по коду
4. **Обработка старых кодов** (до 2018) — конкретный алгоритм маппинга `5B...` → `6B...` / `7M...`
5. **Gotchas** — задокументированные ловушки и типовые ошибки валидации ЕПВО
6. **JSON-примеры** полных пайлоадов для каждого шага pipeline

## 4. Scope

Однофазная задача. Один RF-документ. Без кода.

### Сущности в scope:

| Сущность | Приоритет | Обоснование |
|----------|:---------:|-------------|
| `PROFESSION` (ГОП) | 🔴 | Корневая: студенты ссылаются через `professionId` |
| `SPECIALIZATIONS` (ОП) | 🔴 | Привязана к `PROFESSION`. Код ОП — ключ для отчётности |
| `PROFESSION_CAFEDRA` | 🟡 | Связь ГОП↔Кафедра. Важна для отчётов СУР |
| `PROFESSION_COST` | 🟡 | Стоимость обучения. Важна для финансового модуля |
| `TYP_CURRICULUM` | 🟢 | Учебные планы. Опционально для Stage 1 интеграции |
| `EDUCATION_PROGRAMS_CODE` | 🟢 | Коды аккредитованных ОП. Опционально |
| `Q_EXAMINATIONS` | 🟢 | Типы гос. аттестации. Связь с выпускниками (TFW-6) |

### Out of scope:
- Код интеграции (Python/Java)
- Специфика конкретного ВУЗа (структура его БД)
- Модуль «Подача заявок» (APPLICATIONS)

## 5. Definition of Done (DoD)

- ✅ 1. RF содержит маппинг полей для `PROFESSION` и `SPECIALIZATIONS` с бизнес-правилами (`IF/ELSE`)
- ✅ 2. Sequencing Contract описан: порядок UPSERT с обоснованием (FK-зависимости)
- ✅ 3. Алгоритм Lookup `centerProfId` через `CenterProfession` задокументирован с примером запроса
- ✅ 4. Обработка старых кодов специальностей (до 2018) описана конкретным алгоритмом
- ✅ 5. ≥5 Gotchas с источниками (PDF, чат, практика)
- ✅ 6. JSON-примеры для PROFESSION и SPECIALIZATIONS (полный пайлоад, готовый к отправке)
- ✅ 7. Маппинг для вспомогательных сущностей (PROFESSION_CAFEDRA, PROFESSION_COST) — хотя бы табличный

## 6. Definition of Failure (DoF)

- ❌ 1. Маппинг описан без конкретных правил (только перечень полей, как в RF_TFW-1.2)
- ❌ 2. Нет инструкции по `CenterProfession` lookup — интегратор не знает, как получить `centerProfId`
- ❌ 3. Sequencing не описан, или описан неправильный порядок (ОП до ГОП)

**При провале:** Вернуть на доработку с требованием формализовать правила и добавить примеры.

## 7. Принципы

1. **Нейтральность к ИС** — инструкция не привязана к конкретной ИС ОВПО. Описываем «что отправить в ЕПВО», а не «откуда взять данные»
2. **Business Logic First** — формулировать правила как `IF/ELSE` псевдокод, не как SQL/Python
3. **Полнота > краткость** — лучше документировать 20 полей с пояснениями, чем 5 без контекста
4. **Ссылки на источники** — каждое нетривиальное правило должно иметь ссылку на PDF (adm_doc.txt, строка N) или чат ЕПВО
