# HL — TFW-7: Процедура удаления приказов из ЕПВО

> **Дата**: 2026-03-18
> **Автор**: Coordinator (AI)
> **Статус**: ✅ HL — Утверждён

---

## 1. Видение

Описать **полную процедуру удаления приказов и связанных сущностей** из ЕПВО через API, чтобы интегратор любой ИС ОВПО мог безопасно откатить ошибочно отправленные приказы.

> Удаление приказа — не атомарная операция. Приказ состоит из дерева 4–7 сущностей (`ORDERS`, `ORDER_SECTIONS`, `SECTION_PERSON`, `ORDER_STUDENT_INFO`, `ORDERS_ADDITIONAL`, `SCHOLARSHIP`, `TRANSCRIPT`), каждая из которых удаляется отдельным вызовом API. Ошибка в порядке или пропуск узла приведёт к «осиротевшим» записям.

### 2.2 Открытые вопросы (требуют исследования / проверки)

1. **Каскадное удаление:** Удаляет ли удаление `ORDERS` автоматически дочерние сущности (`ORDER_SECTIONS`, `SECTION_PERSON` и т.д.), или каждую нужно удалять отдельно?
2. **Тип composite key:** Для `ORDERS` — `UNIVERSITY_ID_COMPOSITE_KEY` (поле `id`) или `ORDER_ID_COMPOSITE_KEY` (поле `orderId`)?
3. **Побочные эффекты:** При удалении приказа о зачислении — нужно ли вручную обновить `STUDENT.status` обратно?
4. **Идемпотентность:** Что возвращает API при попытке удалить несуществующую запись?
5. **Мягкое удаление (soft delete):** Записи удаляются физически или помечаются как неактивные? Влияет ли это на повторное создание с тем же ID?

## 4. Фазы

Задача **однофазная** — исследовательская документация (RF), не требующая изменения кода.

### Единая фаза: Документирование процедуры удаления 🔴

Deliverables:
- Pipeline удаления для основных типов приказов (type=2, 3, 4, 6, 7, 10, 13, 14, 27, 28, 30, 31, 34)
- **Детализированный pipeline для type=13** (стипендии) — 7 сущностей, включая SCHOLARSHIP и TRANSCRIPT
- Таблица composite key → body-поле для каждой сущности приказа (8+ сущностей)
- Побочные эффекты на STUDENT после удаления
- Ограничения и особенности (каскадность, идемпотентность, soft delete)
- Практические наблюдения из AD ЕПВО: ID-формулы, month_offset, порядок сущностей
- Список открытых вопросов, разрешённых или неразрешённых

## 5. Definition of Done (DoD)

- ✅ 1. RF содержит pipeline удаления для приказов (обратный порядок сущностей относительно pipeline создания из RF_TFW-5)
- ✅ 2. Для каждой сущности указан composite key type и поле ID в body запроса `/delete`
- ✅ 3. Описаны побочные эффекты: какие поля `STUDENT` и вспомогательных сущностей (`RETIRES`, `SCHOLARSHIP`) нужно обновить после удаления
- ✅ 4. Зафиксированы ограничения API (каскадность, идемпотентность, soft delete)
- ✅ 5. Открытые вопросы из §2.2 разрешены или явно помечены как ❓ (требующие практической верификации)

## 6. Definition of Failure (DoF)

- ❌ 1. Не установлен порядок удаления (дочерние перед родительскими или наоборот)
- ❌ 2. Composite key types указаны неверно (400/500 от API при удалении)

**При провале:** Зафиксировать в Observations; провести практическую верификацию через API.

## 7. Принципы

1. **Обратный порядок** — pipeline удаления = зеркальное отражение pipeline создания из RF_TFW-5
2. **Нейтральность** — документация для любой ИС ОВПО, без привязки к конкретному вузу или ИС
3. **Честность о пробелах** — неверифицированные гипотезы помечаются ⚠️, неизвестные — ❓
4. **Привязка к источникам** — каждый факт со ссылкой на RF/KNOWLEDGE/chat
