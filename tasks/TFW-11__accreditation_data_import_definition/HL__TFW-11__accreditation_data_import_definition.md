## 1. Vision
Определить структуру и реализовать механизм получения официальных данных об аккредитации из системы ЕПВО для их отображения в локальных системах университета (dpdb, ИС ОВПО). Данные, сформированные аккредитационными агентствами в ЕПВО, должны стать "источником истины" для пользователей вуза.

**Impact:** Пользователи ИС ОВПО видят актуальные и официальные данные об аккредитации ОП и вуза без необходимости заходить в ЕПВО.

> "Теперь у нас в системе всегда актуальные данные по аккредитации, напрямую из реестра ЕПВО."

## 2. Current State (As-Is)
*   **Источник данных:** ЕПВО (данные вносятся аккредитационными агентствами).
*   **Локальные системы:** В `dpdb` и ИС ОВПО отсутствуют или не синхронизированы представления этих данных.
*   **Проблема:** Университет не генерирует эти данные, поэтому требуется односторонний импорт/синхронизация "сверху вниз".

## 3. Target State (To-Be)
Определена структура данных и реализован процесс импорта сущностей `ACCREDITATION_AGENCIES` и `UNIVERSITY_ACCREDITATION` (и других смежных) в локальную БД.

### 3.1 Result Visualization

**EPVO API (Источник):**
JSON со списком агентств и записей об аккредитации вуза/программ.

**Local DB / ИС ОВПО (Результат):**
Представление (View) или таблицы, содержащие:
- Наименование агентства (RU/KZ/EN)
- Тип аккредитации (институциональная/специализированная)
- Сроки действия (дата начала/окончания)
- Номер сертификата
- Статус в реестре

### 3.2 Value Flow
`EPVO API (OrgData) → Python Sync Script → Local DB (dpdb) → ИС ОВПО UI`

## 4. Phases
🔴 Phase A: Исследование структуры данных в ЕПВО и проектирование локального представления.
🟡 Phase B: Реализация скрипта импорта и создание SQL-представлений в `dpdb`.

### Phase A: Исследование и проектирование 🔴
- Анализ полей сущностей `ACCREDITATION_AGENCIES` и `UNIVERSITY_ACCREDITATION` в ЕПВО.
- Проверка наличия данных по аккредитации ОП (в `SPECIALIZATIONS` или отдельных таблицах).
- Проектирование структуры таблиц/представлений в `dpdb` для хранения этих данных.

## 5. Definition of Done (DoD)
- ✅ 1. Документирована полная структура данных об аккредитации, доступная через API ЕПВО.
- ✅ 2. Спроектирована схема таблиц/представлений для `dpdb`.
- ✅ 3. Реализован базовый скрипт импорта данных из ЕПВО в `dpdb`.
- ✅ 4. Данные об аккредитации доступны для отображения в ИС ОВПО.

## 6. Definition of Failure (DoF)
- ❌ 1. Несоответствие данных в ЕПВО и локальной копии из-за ошибок маппинга.
- ❌ 2. Отсутствие важных полей (например, дат) в API, которые видны в веб-интерфейсе ЕПВО.

**On failure:** Сверить данные через Swagger API ЕПВО, уточнить у техподдержки наличие расширенных полей.

## 7. Principles
1. **Source of Truth** — данные ЕПВО являются приоритетными.
2. **Read-Only Local** — локальные таблицы для этих данных должны быть доступны только для чтения (через синхронизацию), так как вуз их не редактирует.

## 10. RESEARCH Case

### Blind Spots
- Мы не знаем точный состав полей в `UNIVERSITY_ACCREDITATION` (обязательность дат, номера сертификатов).
- Неясно, как связаны аккредитации конкретных ОП с общим списком аккредитаций (через FK или отдельные сущности).

### Hypotheses
| # | Hypothesis | Status |
|---|----------|--------|
| H1 | Сущность `UNIVERSITY_ACCREDITATION` содержит как институциональную, так и программную аккредитацию. | open |
| H2 | Агентства заполняют данные на трех языках, и API их корректно отдает. | open |

### Risks of Not Researching
Неправильное проектирование локальной БД может привести к потере важных метаданных (например, статуса аккредитации или области действия).

### Proposed RESEARCH Focus
1. **Gather**: Выполнить тестовые запросы к `ACCREDITATION_AGENCIES` и `UNIVERSITY_ACCREDITATION` через Swagger или `epvo_client.py`.
2. **Extract**: Зафиксировать структуру JSON-ответов.
3. **Challenge**: Проверить, достаточно ли этих данных для формирования полноценного представления в ИС ОВПО.
