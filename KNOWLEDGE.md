# EPVO Knowledge Base (KNOWLEDGE.md)

> **Цель:** Сборник неявных правил, требований и особенностей работы с Единой платформой высшего образования (ЕПВО), выявленных в процессе интеграции и анализа работы техподдержки.

## 1. Основные принципы API

### 1.1. Механизм UPSERT (`POST /org-data/list/save`)
- **Единственный эндпоинт:** Все сохранения и обновления осуществляются через `/org-data/list/save`. Одиночный `/org-data/save` не работает (404 Not Found).
- **Массивы данных:** Данные всегда передаются в виде массива JSON-объектов, даже если отправляется одна запись.
- **Логика работы (UPSERT):** На стороне сервера ЕПВО по композитному ключу (см. ниже) определяется логика: если запись с таким ключом уже существует — она обновляется (перезаписывается), если нет — создается новая.
- **⚠️ Полная перезапись (Full Replace), не Partial Update:** При обновлении записи **все поля** должны быть переданы повторно. Если поле не включено в JSON — оно будет **сброшено в `null`**. Частичное обновление (передача только изменённых полей) **не поддерживается**. Источник: чат ЕПВО, Бахтияр 26.06.2023 («при отправке save должны присутствовать все поля»); подтверждено кейсом Евгения 18.09.2023 (поля `scholarshipPayDate` и `scholarshipMoney` «остались null после отправки»).
- **Частичный успех (Partial Success):** Если часть записей валидна, а часть нет, сервер вернёт 200 OK, сохранит валидные записи, а информацию по ошибочным вернет в массиве `failedRecords`.

### 1.2. Композитные ключи (Composite Keys)
Для каждого справочника механизм UPSERT опирается на свой идентификатор:
- `STUDENT`, `STUDENT_INFO`, `STUDENT_DIPLOMA_INFO` -> `studentId`
- `FACULTIES` -> `facultyId`
- `CAFEDRA` -> `cafedraId`
- `TUTOR` -> `tutorId`
- `GROUPS` -> `groupId`
- `PROFESSION` -> `professionId`

### 1.3. Авторизация и подключение (Credentials)
- **Базовый URL API:** `https://epvo.kz/isvuz/api`
- **Аутентификация:** Используется `Basic Auth`. В заголовок `Authorization` передается закодированная в Base64 строка формата `username:password`.
- **Логин (username):** ID ВУЗа (например, `[ID_ВУЗА]`).
- **Пароль (password):** Сгенерированный пароль для интеграции (например, `[YOUR_PASSWORD]`).
*(Креды взяты из `.env` смежного проекта `AD ЕПВО`)*

### 1.4. Ограничения запросов (Query Limitations)
- **Нет серверной фильтрации:** Эндпоинты `find-all-pageable` возвращают **все** записи справочника для данного `universityId`. Фильтрация по произвольным полям (например, по `studentId` в `ORDER_STUDENT_INFO`) **не поддерживается** — только постраничная выдача.
- **`find-by-id`** ищет ровно одну запись по её **композитному ключу** (например, `orderId` для `ORDERS`), а не по FK-полям.
- **Следствие:** Для получения, например, приказов конкретного студента необходимо:
  1. Выгрузить все записи `ORDER_STUDENT_INFO` через `find-all-pageable`.
  2. Отфильтровать локально по `studentId`.
  3. По найденным `sectionId` извлечь `ORDER_SECTIONS` → `orderId` → `ORDERS`.
- **Общий паттерн:** Любой поиск по FK-полям = **полная выгрузка + локальная фильтрация**.

### 1.5. Проверенные эндпоинты (Validated Endpoints)

Результаты практической проверки API (2026-03-12):

| TypeCode | Endpoint | Результат | Примечание |
|----------|----------|:---------:|------------|
| `STUDENT` | `/org-data/STUDENT/find-all-pageable` | ✅ 200 | 65 655 записей |
| `ORDERS` | `/org-data/ORDERS/find-all-pageable` | ✅ 200 | 33 899 приказов |
| `ORDER_CATEGORY` | `/org-data/ORDER_CATEGORY/find-all-pageable` | ❌ 400 | Не поддерживается, несмотря на наличие в OpenAPI spec |

>
> **Следствие:** Значения `orderTypeId` для типов приказов (зачисление, отчисление, академ. отпуск, продление и т.д.) можно получить только **косвенно** — анализируя поле `orderTypeId` в уже существующих записях `ORDERS`.

### 1.6. File API (загрузка и скачивание файлов)

ЕПВО предоставляет отдельный слой API для работы с файлами (сканы приказов, сертификаты и т.д.).

#### Загрузка: `POST /v1/file/upload`
- **Content-Type:** `multipart/form-data`
- **Параметры:** `file` (binary, required), `universityId` (int, required), `filename` (string, optional)
- **Ответ (200):** JSON с `fileId` (UUID), `fileName`, `size`, `mimeType`

#### Скачивание: `GET /v1/file/{fileId}`
- **Path:** `fileId` (string/UUID, required)
- **Ответ (200):** бинарный поток файла с заголовками `Content-Type` и `Content-Disposition`

#### Привязка файлов к OrgData
1. Загрузить файл → получить `fileId` из ответа.
2. Передать `fileId` в нужное поле при сохранении OrgData записи через `/org-data/list/save` (например, `orders.fileId`, `employee_orders.fileName`).
3. Для получения `fileId` уже загруженного файла — выгрузить запись через `find-all-pageable` или `find-by-id`.

> ⚠️ **Не верифицировано:** Точный путь может быть `/v1/file/upload` или `/api/file/upload` — в чате техподдержки прямых упоминаний URL не найдено. Требуется практическая проверка. Источник: теоретическая документация `RF_TFW-2.7__file_api.md` на основе OpenAPI spec.

## 2. Строгие правила валидации (на основе базы ошибок)

### 2.1. Гражданство (Citizenship)
- **НЕЛЬЗЯ передавать `null`:** Это самая частая причина 500/400 ошибок при интеграции студентов (`STUDENT`, `STUDENT_INFO`) и преподавателей (`TUTOR`).
- **Связь с ИИН:** Если у физического лица (иностранца) нет ИИН, он всё равно обязан быть в базе, и его `citizenshipId` (гражданство по паспорту) становится критически важным для идентификации.
- Формат: Ожидается корректный ID из системного справочника `CenterCountry` (например, Казахстан — 113, иногда в старых базах 398 — нужно сверяться со справочником ЕПВО).

### 2.2. Соответствие справочникам ОП и ГОП
- Поле `professionsid` в таблице студентов должно строго соответствовать полю `professionid` в локальной таблице профессий (`professions`).
- **Глобальная привязка:** Локальное поле `code` из таблицы `professions` должно совпадать с `code` из центрального справочника ЕПВО `center_profession`.
- Несовпадение этих кодов приводит к ошибке «Имеются обучающиеся, не попавшие в отчет из-за некорректных данных».

### 2.3. Синхронизация приказов и статусов
- Нельзя передавать студента со статусом «обучающийся» (`studyStatusId`), если по нему есть проведённый приказ об отчислении или выпуске.
- Студент не может попасть в отчет на стипендию/финансирование, если у него отсутствует базовый приказ о зачислении.
- Если студент находится в академическом отпуске, обязательно должны быть переданы сканы/данные подтверждающих документов.

### 2.4. Мультиязычность в справочниках (SUBJECT, FACULTIES и др.)
- Поля `nameRu`, `nameKz`, `nameEn` (или `briefInfo...`) **обязательны к заполнению** для всех трех языков. При попытке обновления, если одно из полей пустое (или содержит null), API вернет ошибку вида "Не заполнено на 3х языках" или выбросит 500 Internal Server Error.

### 2.5. Настройки формы обучения (studyforms) и генерация дипломов
При генерации номеров дипломов строгие проверки идут по полям сроков (во избежание ошибки кол-ва освоенных кредитов):
- `trainingCompletionTerm` (семестр завершения) **не должно быть больше** `termsCount` (кол-во семестров в году). Обычно макс. значение `termsCount` = 4.
- `courseCount` (кол-во курсов) не должно превышать 5.

### 2.6. Приказы об отчислении и движении контингента
- **Статус студента:** Нельзя просто отправить приказ об отчислении, оставив статус активным (обучающийся). Необходимо передать измененный статус (`studyStatusId` = Отчислен) в карточке `STUDENT`, иначе система усмотрит конфликт и выдаст ошибку.
- **Тип и категория приказа:** `orderTypeId` и `categoryId` — обязательные поля (в чате ЕПВО техподдержка часто указывает на их отсутствие: «В приказах по отчислению не заполнена категория»). Тип отчисления должен соответствовать значению системного справочника (чаще всего `orderTypeId = 3`).
- **Дата движения (`movement_date` / `dateOfEmployment`):** Самое важное поле для расчета финансов. Месяц прекращения выплаты стипендии/ГосГранта считается по фактической дате движения (например: дата приказа 11.10.2023, но отчислен с 20.10.2023 — датой окончания финансирования в отчете будет 20 октября).
- **Перевод в другой ВУЗ:** При отчислении переводом в другой/зарубежный ВУЗ необходимо передавать сущность `ORDERS_ADDITIONAL` и обязательно указывать системный идентификатор (ID) принимающего ВУЗа (`toCenterUniversityId` или аналогичное поле связи).

### 2.7. Проверка ИИН (7-й разряд и пол)
- ЕПВО строго проверяет соответствие 7-й цифры ИИН полу (3/5 для мужчин, 4/6 для женщин и т.д.).
- **Иностранцы:** Для иностранных граждан (у кого `citizenshipId != 113`) проверка 7-й цифры **отключена**.
- **Казахстанцы с нестандартным ИИН:** Если ИИН выдан госорганом РК с ошибкой в 7-й цифре, и он не соответствует полу из паспорта, нужно установить флаги обхода валидации:
  - Для **обучающихся** (в `STUDENT`): передать `incorrectiin = true`.
  - Для **выпускников** (в сущности выпускников): передать `iin_gived_by_government = true`.

### 2.8. Академический отпуск (Academic Leave / `retires`)

#### Типы приказов
В модуле «Финансирование» определены три типа приказов по академическому отпуску (соответствующие значения `orderTypeId` из справочника `OrderType`):
1. **Приказ об академическом отпуске** — уход в отпуск.
2. **Приказ о продлении академического отпуска** — продление сроков.
3. **Приказ о выходе из академического отпуска** — возврат к обучению.

#### Таблица `retires` (обязательна для передачи)
При оформлении/продлении/завершении академического отпуска необходимо передавать запись в таблицу `retires`:

| Поле | Тип | Обяз. | Описание |
|------|-----|:-----:|----------|
| `university_id` | int | ✅ | ID ОВПО |
| `id` | int | ✅ | Уникальный ID записи |
| `studentid` | int | ✅ | ID обучающегося (`→ students`) |
| `startyear` | int | ✅ | Учебный год начала отпуска |
| `startterm` | int | ✅ | Номер акад. периода начала отпуска |
| `startcourse` | int | ✅ | Номер курса начала отпуска |
| `termscount` | int | ✅ | Количество акад. периодов отпуска |
| `startDate` | date | ✅ | Дата начала отпуска |
| `finishDate` | date | ✅ | Дата окончания отпуска |
| `finished` | bool | — | `true` — есть приказ о выходе |
| `orderid` | int | УО | ID приказа о выходе (обяз., если `finished=true`) |
| `retire_reason` | int | УО | Причина отпуска (обяз. для категорий «По здоровью» / «Рождение ребёнка») |
| `inability_start_date` | date | УО | Дата начала справки о нетрудоспособности (обяз., если причина «беременность и роды») |
| `inability_end_date` | date | УО | Дата окончания справки о нетрудоспособности |

**Полная таблица маппинга причин академ отпуска (из чата техподдержки, 10.11.2025):**

Значение, отображаемое в карточке финансирования, определяется **комбинацией** типа приказа (`orderid`), категории приказа (`categoryId`) и поля `retires.retire_reason`:

| # | Причина в финансировании | orderid | categoryId | retire_reason |
|---|--------------------------|---------|------------|:-------------:|
| 1 | Беременность и роды | 6 | 17 (рождение/удочерение/усыновление) | `1` |
| 2 | Усыновление/удочерение ребенка | 6 | 17 | `2` |
| 3 | Туберкулёз | 6 | 16 (по состоянию здоровья) | `1` |
| 4 | Другие медицинские показания | 6 | 16 | `2` |
| 5 | В связи с призывом на воинскую службу | 6 | 15 (призыв на воинскую службу) | — |
| 6 | По состоянию здоровья | 6 | 16 | *пустое* |
| 7 | При рождении/усыновлении/удочерении ребенка | 6 | 17 | *пустое* |
| 8 | Приказ о продлении академ. отпуска | 14 | — | *пустое* |
| 9 | По уходу за ребенком до 3 лет | 6 | 17 | `3` |

> ⚠️ **Частая проблема:** Если `retires.retire_reason` не заполнено (или заполнено некорректно), причина академ отпуска **не отображается** в карточке финансирования студента, хотя сам приказ с `categoryId` передан верно. Источник: чат ЕПВО, сообщение Бахтияра от 10.11.2025 17:03.

#### Флаг `students.isinretire`
- Используется в отчётах СУР и Адм. отчётах для фильтрации контингента.
- `0` — студент **не** в академическом отпуске; `1` — находится в отпуске.
- При продлении отпуска значение остаётся `1`.

#### Логика продления
1. Создать приказ (`ORDERS`) типа «Продление академического отпуска» (`orderid=14`).
2. Привязать студента через `ORDER_STUDENT_INFO` или `ordersadditional`.
3. **Обновить существующую** запись в `retires` (не создавать новую!): увеличить `termscount` и/или сдвинуть `finishDate`.
4. Передать подтверждающие документы (сканы — обязательное требование ЕПВО).
5. При UPSERT обязательно передавать **все поля** записи `retires`, включая `reasonBody: ""`, `filename`, `mimetype` — иначе они сбросятся в `null` и запись перестанет корректно отображаться.

> ⚠️ **Связь приказа продления с исходным приказом:** Прямого FK-поля (`parentOrderId` и т.п.) между приказами нет. Связь осуществляется **косвенно** — через одну и ту же запись `retires` (по `studentid`). При продлении обновляется **та же** запись, а не создаётся новая (дубли вызывают ошибку). Источник: чат ЕПВО, Бахтияр 31.01.2024 и Medet Turdaliyev 15.12.2023.

#### Удаление записей `retires`
Удаление возможно через стандартный эндпоинт OrgData:
```
POST /org-data/RETIRES/delete
```
В тело запроса передаётся композитный ключ записи.

**Важно:** Удаление из `retires` **недостаточно** для снятия статуса академ отпуска:
1. Необходимо также установить `students.isinretire = 0`.
2. Все связанные приказы (об отпуске и о выходе) должны быть согласованы.
3. Отчёты могут **кешироваться** — изменения не всегда отражаются мгновенно.

> ⚠️ **Случай из практики (чат, 13.10.2023, Astana IT University):** После удаления записи из `retires` и установки `isinretire = false` у студента **остался «доп статус академ отпуск» в отчёте**. Причина: наличие приказов об отпуске без корректного завершения цепочки (приказ о выходе был, но система продолжала учитывать доп статус в кешированном отчёте).

#### Обновление дат в приказах об академическом отпуске
Изменение дат в приказах технически возможно через стандартный UPSERT (`/org-data/list/save`). Однако, поскольку UPSERT работает как **полная перезапись** (см. раздел 1.1), при обновлении любого поля (включая даты) необходимо **повторно передать все поля** записи.

> ⚠️ **Случай из практики (чат, 08.11.2023):** Babur Rustauletov сообщил: «После обновления приказа по академическому отпуску у студента **исчезли подтверждающие документы**». Причина: при UPSERT не были повторно переданы связи с файлами, и они были сброшены в `null`.

**Рекомендации:**
1. Перед обновлением — сделать `find-by-id` для получения текущего состояния записи.
2. Изменить нужные поля (даты) в полученном JSON.
3. Отправить **полный** объект через `save`, включая все существующие данные.
4. Проверить, что прикреплённые документы не были утеряны.

#### Переключение `finished: true → false` (возврат в академотпуск)
Изменение `retires.finished` с `true` на `false` технически возможно через UPSERT, но несёт **риски рассинхронизации**:

**Потенциальные проблемы:**
1. **Рассинхронизация `finished` и `isinretire`:** Если обновить `finished=false`, но не вернуть `students.isinretire=1`, система будет видеть противоречие — в `retires` студент в отпуске, а в `students` — нет. Именно такая рассинхронизация приводила к «залипшему» доп. статусу (чат, Ersultan, 13.10.2023).
2. **Поле `orderid` в `retires`:** При `finished=true` содержит ID приказа о выходе. Переключение на `false` без очистки `orderid` создаёт логическое противоречие (выход не завершён, но приказ о выходе есть).
3. **Выпадение из стипендиальных отчётов:** При `finished=false` студент считается «в отпуске» и может быть **исключён из отчётов финансирования** (Бахтияр, 23.09.2023: «если `isinretire` = `null`, стипендиат не отобразится в отчете»).
4. **Full Replace:** При обновлении `finished` через UPSERT нужно передать **все поля** `retires`, иначе остальные сбросятся в `null` (см. раздел 1.1).

**Корректная процедура возврата в академотпуск:**
1. Обновить `retires`: `finished=false`, очистить `orderid` (приказа о выходе).
2. Обновить `students.isinretire = 1`.
3. Убедиться, что приказ о выходе удалён или деактивирован.
4. При UPSERT передать **все поля** обеих записей (`retires` и `students`).

> **Источник:** `instructions/ОВПО_Инструкция_для_передачи_данных_модуль_Финансирование_02092024.txt`, строки 15–17, 183–266; `adm_doc.txt` (поле `students.isinretire`); `sur_doc.txt`; чат ЕПВО.

### 2.9. Стипендии (`SCHOLARSHIP`) — полная практическая схема

**typeCode:** `SCHOLARSHIP` | **Composite Key:** `UNIVERSITY_ID_COMPOSITE_KEY` → `{ type, id }`

Одна запись = один месяц выплаты стипендии одному студенту.

#### Полная таблица полей

| Поле | Тип | Обяз. | Описание |
|------|-----|:-----:|----------|
| `typeCode` | string | ✅ | Всегда `"SCHOLARSHIP"` |
| `universityId` | int | ✅ | ID вуза |
| `id` | int | ✅ | Уникальный ID записи стипендии |
| `studentId` | int | ✅ | ID обучающегося (→ `STUDENT`) |
| `scholarshipYear` | int | ✅ | Год **выплаты** |
| `scholarshipMonth` | int | ✅ | Месяц **выплаты** (1–12) |
| `scholarshipPayDate` | date | — | Дата фактической выплаты |
| `scholarshipMoney` | double | ⚠️ | Сумма стипендии. Без этого поля — **500 ошибка** |
| `scholarshipTypeId` | int | ✅ | Тип стипендии из центрального справочника (значения могут меняться! ID=1 временно удалялся 19.09.2023) |
| `terminationDate` | date | — | Дата прекращения стипендии (при лишении) |
| `additionalPayment` | bool | — | Доплата (`true`/`false`) |
| `sectionId` | int | ✅ | ID секции приказа о **назначении** (→ `ORDER_SECTIONS` → `ORDERS`) |
| `scholarshipAwardYear` | int | ✅ | Учебный год, по итогам которого **назначена** (обязательно по инструкции от 02.09.2024) |
| `scholarshipAwardTerm` | int | ✅ | Семестр, по итогам которого **назначена** (1 или 2) (обязательно по инструкции) |
| `overallPerformance` | int | ✅ | Успеваемость: 5=отлично, 4=хорошо, 3=удовл., 2=неудовл., 1=без учёта, 0=нет данных (обязательно по инструкции) |
| `termination_ord_sectionid` | int | — | ID секции приказа о **прекращении** стипендии |

#### Практический JSON-пример

```json
{
  "typeCode": "SCHOLARSHIP",
  "universityId": 7,
  "id": 1,
  "studentId": 44386,
  "scholarshipYear": 2022,
  "scholarshipMonth": 1,
  "scholarshipPayDate": "2023-09-26",
  "scholarshipMoney": 41898.0,
  "scholarshipTypeId": 1,
  "terminationDate": null,
  "additionalPayment": false,
  "sectionId": 25515,
  "scholarshipAwardYear": 2022,
  "scholarshipAwardTerm": 2,
  "overallPerformance": 4,
  "termination_ord_sectionid": 0
}
```

#### Ключевые нюансы (из чата ЕПВО)

1. **`scholarshipMoney` де-факто обязательно** — Platonus заявлял, что поле «не нужно», но без него сервер возвращает 500. При этом поле **не возвращается** при `find-all-pageable` (записывается, но не читается обратно). Финцентр использует его для расчётов. Источник: чат ЕПВО, Евгений (universityId=15), 18.09.2023.
2. **`sectionId`** — ссылка на секцию приказа о назначении стипендии. Цепочка: `sectionId` → `ORDER_SECTIONS` → `orderId` → `ORDERS`.
3. **`scholarshipAwardYear/Term` vs `scholarshipYear/Month`** — первая пара = период *назначения* (по итогам какого семестра), вторая = период *выплаты*. Поля `Award*` не описаны в документации Platonus, но при `null` данные о назначении будут неполными.
4. **`overallPerformance`** — числовой код успеваемости (не GPA и не балл).
5. **При лишении стипендии:** `scholarshipTypeId` остаётся тем же, `scholarshipMoney = 0`, `overallPerformance = 2 или 3`. Нужен приказ о лишении → его `sectionId` идёт в `termination_ord_sectionid`.

> Источник: `chat_epvo.json`, сообщения участников universityId=7 и universityId=15, период 09.2023–01.2025; шаблон из сообщения от 20.01.2025; инструкция ОВПО от 02.09.2024.

### 2.10. Предусловия отображения в карточке финансирования

Чтобы данные по стипендии были видны в карточке финансирования ЕПВО, **все** условия должны быть выполнены:

1. **`STUDENT.isinretire`** = `0` или `1` (НЕ `null` — иначе студент не отобразится). Источник: чат, Бахтияр 23.09.2023.
2. **`STUDENT.enrollOrderDate`** — не из текущего семестра для не-первокурсников. Источник: чат, Евгений 21.09.2023.
3. **`STUDENT_INFO.iic/bic`** — заполнены для грантников.
4. **`ORDERS`** с `orderTypeId=13` загружен, `categoryId` заполнен.
5. **`ORDER_SECTIONS`** — приказ имеет секцию.
6. **`ORDER_STUDENT_INFO`/`sectionpersons`** — студент привязан к секции, `movement_date` и `overall_performance` заполнены.
7. **`SCHOLARSHIP`** — записи созданы с корректным `sectionId` (ID **секции**, НЕ `orderId` приказа), `scholarshipMoney`, `scholarshipTypeId`, `scholarshipAwardYear`, `scholarshipAwardTerm`, `overallPerformance`.
8. **Кэш ЕПВО** — задержка ~1–2 часа после 200 OK.

> Источник: официальная инструкция ОВПО от 02.09.2024; чат ЕПВО, диалоги 19–23.09.2023.

## 3. Решение частых проблем

- **Ошибка 400 Bad Request:** Обычно означает синтаксическую ошибку в JSON, неверные типы данных, пустые обязательные поля (как `citizenshipId`) или неверно указанный `typeCode`.
- **Ошибка 500 Internal Server Error:** Часто сервер ЕПВО маскирует внутренние `NullPointerException` или проблемы с валидацией БД под 500 ошибкой (например, отправлен `null` в поле `scholarshipMoney`, или отсутствует перевод названия дисциплины). В таких случаях нужно перепроверять весь DTO на наличие `null` в обязательных полях.
- **Ошибки авторизации 401 / "Вход недоступен":** Если проблема возникает массово, может лежать сервер или истек пароль. Локально помогает сброс кэша браузера.
- **Карточка финансирования не обновляется:** Изменения в `retires` и `orders` отражаются в карточке финансирования с **задержкой ~1-2 часа** (кэш на стороне ЕПВО). Не стоит отправлять повторные запросы, если API вернул 200 — подождать. Источник: практический опыт, 2026-03-13.
- **Дублирование записей RETIRES:** При продлении академ. отпуска **нельзя** создавать новую запись `retires` с другим `id` для того же студента. Нужно обновлять существующую запись (сохраняя оригинальный `id`). Дубликаты могут привести к конфликтам в карточке финансирования. Для удаления мусорных записей: `POST /org-data/RETIRES/delete` с `{"type": "UNIVERSITY_ID_COMPOSITE_KEY", "id": <retires_id>}`.

### 2.11. Выпускники (`GRADUATES`) — отдельная сущность и порядок отправки

#### Сущность GRADUATES
`GRADUATES` — это **отдельная сущность** (не обновление `STUDENT`), содержащая полный профиль выпускника. `typeCode: "GRADUATES"`, композитный ключ: `GRADUATE_ID_COMPOSITE_KEY` → `{ type, graduateId }`.

> ⚠️ **Самая «строгая» сущность API:** ~20 обязательных (not null) полей, включая диплом, приказы, персональные данные, ГОП/ОП, GPA.

#### Фильтрация: кто попадает в GRADUATES
```text
students.isstudent = 3  AND  students.coursenumber = 0
AND  graduates.degree_id NOT IN (0, 10)  // Не подготовительное отделение
AND  есть запись в таблице graduates
```

#### Статус студента: `status` vs `isStudent`
| Контекст | Поле | Значения |
|----------|------|----------|
| ОСМС-отчёт | `isStudent` | `1` — активный, `0` — академ, `3` — неактивный |
| OpenAPI spec (STUDENT) | `status` | `1` — обучается, `2` — абитуриент, `3` — отчислен, `4` — выпускник |

> ⚠️ **Это одно физическое поле.** При выпуске выставляется `status=4` (или `isStudent=3` для ОСМС). Интегратор должен верифицировать, какое значение принимает API конкретного ВУЗа. Источник: `RF_TFW-1.3` (строка 34), `RF_TFW-4.A`, `RF_TFW-4.E`.

#### Обязательный порядок отправки (Sequencing Contract)
При выпуске данные отправляются **строго последовательно**:
1. `TRANSCRIPT` — все оценки за финальную сессию.
2. `ORDERS` (orderType=6) + `ORDER_SECTIONS` + `SECTION_PERSON` + `ORDER_STUDENT_INFO` — приказ о выпуске.
3. `STUDENT` update — `status=4`, `courseNumber=0`.
4. `STUDENT_DIPLOMA_INFO` — серия, номер, дата диплома.
5. `GRADUATES` — полный профиль выпускника.

**Обоснование:** Шаг 1 перед выпуском обязателен — ЕПВО использует транскрипты для GPA. Приказ (шаг 2) регистрируется до смены статуса, иначе orphan-запись. `GRADUATES` последним — ссылается на `studentId` и ожидает статус «Выпускник».

### 2.12. Дипломные данные — три точки ввода

Серия и номер диплома отправляются в **три** сущности:

| Сущность | Поля | Обязательность |
|----------|------|:--------------:|
| `GRADUATES` | `iacDiplomaSeries`, `iacDiplomaNumber`, `diplomaDate` | ✅ not null |
| `STUDENT_DIPLOMA_INFO` | `diplomaSeries`, `diplomaNumber`, `diplomaDate` | — |
| `STUDENT_INFO` | `diplomaSeries`, `diplomaNumber`, `diplomaDate` | — |

> ⚠️ **Рассогласование** между тремя сущностями вызовет конфликт при проверке дипломов на `https://epvo.kz/#/verification_education_document`. Источник данных должен быть **единым**.

**GPA:** Обязателен для поступивших с 2010 года и позже (`enrollYear >= 2010`).

#### Условные поля по degreeId
- **Докторантура** (`degreeId IN (6, 7)`): `doctorDefended`, `defenseOfDissertationInOtherHpeo`, `dateDissertationDefense`.
- **Интернатура** (`degreeId = 4`): `internIacDiplomaSeries/Number`, `internProtocolNumber/Date`, `internRegistrationNumber`, `internDateOfIssue`, `internHonoursDiploma`.

### 2.13. Общежития (`DORMITORY`) — данные для СУР

`typeCode: "DORMITORY"`, композитный ключ: `DORMITORY_ID_COMPOSITE_KEY` → `{ type, dormitoryId }`.

Передаётся для отчёта **ВП-7** (обеспеченность общежитиями) в рамках СУР (октябрь–ноябрь) и Нац. рейтинга (февраль–март).

**Scope ВП-7 — кто попадает в отчёт:**
- Активные студенты (`status=1`).
- Иногородние (`local=false`) или иностранные (`sitizenshipId != 113`).
- Нуждающиеся в общежитии (`dormStateId IN (2, 3)`).
- Учитываются только **открытые** общежития (`finishDate IS NULL`).

Ключевые поля: `beds` (койко-места), `rentalType` (1=собственное, 2=арендованное), `square`, `floorCount`, `roomCount`, доступность для маломобильных (`rampCount`, `liftCount`, `hasSpecialToilets`).

> Источник: `adm_doc.txt` (Таблица 13, раздел ВП-7), `RF_TFW-1.8__infrastructure.md`, `RF_TFW-4.E__graduates.md`.
