## Технические ограничения и правила валидации

### 2.1. Гражданство (Citizenship)
- **НЕЛЬЗЯ передавать `null`:** Это самая частая причина 500/400 ошибок при интеграции студентов (`STUDENT`, `STUDENT_INFO`) и преподавателей (`TUTOR`).
- **Связь с ИИН:** Если у физического лица (иностранца) нет ИИН, он всё равно обязан быть в базе, и его `citizenshipId` (гражданство по паспорту) становится критически важным для идентификации.
- Формат: Ожидается корректный ID из системного справочника `CenterCountry` (например, Казахстан — 113, иногда в старых базах 398 — нужно сверяться со справочником ЕПВО).

### 2.2. Соответствие справочникам ОП и ГОП
- Поле `professionsid` в таблице студентов должно строго соответствовать полю `professionid` в локальной таблице профессий (`professions`).
- **Глобальная привязка:** Локальное поле `code` из таблицы `professions` должно совпадать с `code` из центрального справочника ЕПВО `center_profession`.
- Несовпадение этих кодов приводит к ошибке «Имеются обучающиеся, не попавшие в отчет из-за некорректных данных».

### 2.4. Мультиязычность в справочниках (SUBJECT, FACULTIES и др.)
- Поля `nameRu`, `nameKz`, `nameEn` (или `briefInfo...`) **обязательны к заполнению** для всех трех языков. При попытке обновления, если одно из полей пустое (или содержит null), API вернет ошибку вида "Не заполнено на 3х языках" или выбросит 500 Internal Server Error.

### 2.7. Проверка ИИН (7-й разряд и пол)
- ЕПВО строго проверяет соответствие 7-й цифры ИИН полу (3/5 для мужчин, 4/6 для женщин и т.д.).
- **Иностранцы:** Для иностранных граждан (у кого `citizenshipId != 113`) проверка 7-й цифры **отключена**.
- **Казахстанцы с нестандартным ИИН:** Если ИИН выдан госорганом РК с ошибкой в 7-й цифре, и он не соответствует полу из паспорта, нужно установить флаги обхода валидации:
  - Для **обучающихся** (в `STUDENT`): передать `incorrectiin = true`.
  - Для **выпускников** (в сущности выпускников): передать `iin_gived_by_government = true`.

## 3. Решение частых проблем

- **Ошибка 400 Bad Request:** Обычно означает синтаксическую ошибку в JSON, неверные типы данных, пустые обязательные поля (как `citizenshipId`) или неверно указанный `typeCode`.
- **Ошибка 500 Internal Server Error:** Часто сервер ЕПВО маскирует внутренние `NullPointerException` или проблемы с валидацией БД под 500 ошибкой (например, отправлен `null` в поле `scholarshipMoney`, или отсутствует перевод названия дисциплины). В таких случаях нужно перепроверять весь DTO на наличие `null` в обязательных полях.
- **Ошибки авторизации 401 / "Вход недоступен":** Если проблема возникает массово, может лежать сервер или истек пароль. Локально помогает сброс кэша браузера.
- **Карточка финансирования не обновляется:** Изменения в `retires` и `orders` отражаются в карточке финансирования с **задержкой ~1-2 часа** (кэш на стороне ЕПВО). Не стоит отправлять повторные запросы, если API вернул 200 — подождать. Источник: практический опыт, 2026-03-13.
- **Отчисленные студенты продолжают попадать в отчеты по стипендии:** ВУЗ обязан отправлять полное «движение» по отчисленным. Нельзя просто удалить их из выгрузки. Необходимо передать приказ об отчислении (`type=3`), указать `movementDate` (дата вступления приказа в силу) и сменить профиль в `STUDENT`: `isStudent=3` (отчислен). **Помимо** отправки приказа, требуется **физически удалить** (`delete_entity`) записи `SCHOLARSHIP` за месяцы, в которых стипендия уже не положена (иначе они «виснут» в финансовых отчётах).

  **Правило «15-го числа»** — граница прекращения выплаты стипендии при отчислении:

  | `movementDate` приказа об отчислении | Стипендия за текущий месяц | Действие с `SCHOLARSHIP` |
  |:------------------------------------:|:--------------------------:|:------------------------:|
  | **До 15-го** числа месяца | ❌ Не назначается | Удалить записи за текущий месяц **и все последующие** |
  | **После 15-го** числа месяца | ✅ Выплачивается полностью | Удалить записи только за **последующие** месяцы |

  Пример: студент отчислен 09.01.2024 (`movementDate`) → удалить `SCHOLARSHIP` за `month=1` (январь) и далее. Студент отчислен 18.03.2024 → запись за `month=3` (март) остаётся, удаляются `month=4, 5, 6`.

  > Источник: чат ЕПВО, Айдар Буранбаев 12.01.2024: *«до 15 января если отчисли студента то на январь месяц назначение стипендии не должно быть»*; представитель вуза 15.07.2024: *«в положении указано, что если студент отчисляется после 15 числа, вуз должен выплатить ему полную стипендию за текущий месяц»*.
- **Ошибка / пустое значение по `overallPerformance` (Успеваемость):** 1) Ожидается **код-инт от 0 до 5**, а **не GPA**. 2) Поле должно передаваться строго в camelCase (`overallPerformance`) в `SCHOLARSHIP`. При ошибке (`overalperfomance`) ЕПВО примет JSON, но выкинет валидационную ошибку из-за отсутствия обязательного поля назначения стипендии. Для приказов об отчислении поле `overallPerformance` не требуется, но при жесткой валидации можно слать `1` («без учета») или `0` («нет данных»).
- **Дублирование записей RETIRES:** При продлении академ. отпуска **нельзя** создавать новую запись `retires` с другим `id` для того же студента. Нужно обновлять существующую запись (сохраняя оригинальный `id`). Дубликаты могут привести к конфликтам в карточке финансирования. Для удаления мусорных записей: `POST /org-data/RETIRES/delete` с `{"type": "UNIVERSITY_ID_COMPOSITE_KEY", "id": <retires_id>}`.
- **Дубликат профиля студента (два `studentId`):** Если в ЕПВО создались два профиля одного студента с разными `studentId` (отличающимися, например, на 1 цифру), необходимо удалить ошибочный профиль через каскадное удаление зависимостей:

  **Порядок удаления (каскад «снизу вверх»):**
  1. `SCHOLARSHIP` — стипендии дубля
  2. `TRANSCRIPT` — оценки дубля
  3. `ORDER_STUDENT_INFO` — привязки к приказам
  4. `RETIRES` — записи академ. отпуска
  5. `STUDENT_DIPLOMA_INFO` — дипломные данные
  6. `STUDENT_INFO` — доп. информация
  7. `STUDENT` — сам профиль

  Каждый вызов:
  ```
  POST /org-data/{CODE}/delete
  Body: {"type": "STUDENT_ID_COMPOSITE_KEY", "studentId": <ОШИБОЧНЫЙ_ID>}
  ```
  > ⚠️ **Если API возвращает 400** — возможно, на ошибочный `studentId` нет записей в данной сущности (это нормально, переходить к следующему шагу). Если удаление `STUDENT` не срабатывает — обратиться в техподдержку support.platonus.kz с указанием обоих `studentId` и ИИН (тикет #132931 — прецедент «Дубликаты ИИН студентов»). Источник: `RF_TFW-2.5`, `platonus_tickets_index.md` (тикет #132931), 2026-03-19.

### 2.11. Выпускники (`GRADUATES`) — отдельная сущность и порядок отправки

#### Сущность GRADUATES
`GRADUATES` — это **отдельная сущность** (не обновление `STUDENT`), содержащая полный профиль выпускника. `typeCode: "GRADUATES"`, композитный ключ: `GRADUATE_ID_COMPOSITE_KEY` → `{ type, graduateId }`.

> ⚠️ **Самая «строгая» сущность API:** ~20 обязательных (not null) полей, включая диплом, приказы, персональные данные, ГОП/ОП, GPA.

#### Фильтрация: кто попадает в GRADUATES
```text
students.isstudent = 3  AND  students.coursenumber = 0
AND  graduates.degree_id NOT IN (0, 10)  // Не подготовительное отделение
AND  есть запись в таблице graduates
```

#### Статус студента: `status` vs `isStudent`
| Контекст | Поле | Значения |
|----------|------|----------|
| ОСМС-отчёт | `isStudent` | `1` — активный, `0` — академ, `3` — неактивный |
| OpenAPI spec (STUDENT) | `status` | `1` — обучается, `2` — абитуриент, `3` — отчислен, `4` — выпускник |

> ⚠️ **Это одно физическое поле.** При выпуске выставляется `status=4` (или `isStudent=3` для ОСМС). Интегратор должен верифицировать, какое значение принимает API конкретного ВУЗа. Источник: `RF_TFW-1.3` (строка 34), `RF_TFW-4.A`, `RF_TFW-4.E`.

#### Обязательный порядок отправки (Sequencing Contract)
При выпуске данные отправляются **строго последовательно**:
1. `TRANSCRIPT` — все оценки за финальную сессию (студент ещё в `status=1`, последний курс).
2. **★ Генерация номера диплома** — выполняется оператором ОВПО в веб-кабинете ЕПВО. Студент должен быть в `status=1` на последнем курсе с достаточным количеством кредитов. На выходе получаем серию и номер.
3. `ORDERS` (`orderTypeId=28`) + `ORDER_SECTIONS` + `SECTION_PERSON` + `ORDER_STUDENT_INFO` — приказ о завершении обучения.
4. `STUDENT` update — `status=4`, `courseNumber=0`.
5. `STUDENT_DIPLOMA_INFO` — серия, номер, дата диплома (данные из шага 2).
6. `GRADUATES` — полный профиль выпускника.

**Обоснование:** Шаг 1 обязателен — ЕПВО использует транскрипты для подсчёта кредитов и GPA. Шаг 2 (генерация) возможен **только** пока студент в `status=1` — после смены на `4` он исчезнет из модуля генерации (см. §2.16). Приказ (шаг 3) регистрируется до смены статуса, иначе orphan-запись. `GRADUATES` последним — ссылается на `studentId` и ожидает статус «Выпускник». QR-код для проверки подлинности диплома активируется только после `status=4`. Источник: «Инструкция ИС ЕПВО Генерация номеров документов ОВПО», 2024.

### 2.12. Дипломные данные — три точки ввода

Серия и номер диплома отправляются в **три** сущности:

| Сущность | Поля | Обязательность |
|----------|------|:--------------:|
| `GRADUATES` | `iacDiplomaSeries`, `iacDiplomaNumber`, `diplomaDate` | ✅ not null |
| `STUDENT_DIPLOMA_INFO` | `diplomaSeries`, `diplomaNumber`, `diplomaDate` | — |
| `STUDENT_INFO` | `diplomaSeries`, `diplomaNumber`, `diplomaDate` | — |

> ⚠️ **Рассогласование** между тремя сущностями вызовет конфликт при проверке дипломов на `https://epvo.kz/#/verification_education_document`. Источник данных должен быть **единым**.

**GPA:** Обязателен для поступивших с 2010 года и позже (`enrollYear >= 2010`).

### 2.13. Общежития (`DORMITORY`) — данные для СУР

`typeCode: "DORMITORY"`, композитный ключ: `DORMITORY_ID_COMPOSITE_KEY` → `{ type, dormitoryId }`.

Передаётся для отчёта **ВП-7** (обеспеченность общежитиями) в рамках СУР (октябрь–ноябрь) и Нац. рейтинга (февраль–март).

**Scope ВП-7 — кто попадает в отчёт:**
- Активные студенты (`status=1`).
- Иногородние (`local=false`) или иностранные (`sitizenshipId != 113`).
- Нуждающиеся в общежитии (`dormStateId IN (2, 3)`).
- Учитываются только **открытые** общежития (`finishDate IS NULL`).

Ключевые поля: `beds` (койко-места), `rentalType` (1=собственное, 2=арендованное), `square`, `floorCount`, `roomCount`, доступность для маломобильных (`rampCount`, `liftCount`, `hasSpecialToilets`).

> Источник: `adm_doc.txt` (Таблица 13, раздел ВП-7), `RF_TFW-1.8__infrastructure.md`, `RF_TFW-4.E__graduates.md`.

### 2.14. Специфика докторантуры (Защита диссертации)

#### Выпуск докторантов без защиты
В строгой логике ЕПВО **докторант не может быть выпущен (`status=4`, `GRADUATES`) без защиты диссертации**. Если докторант завершил теоретический курс, но не вышел на защиту, он переводится в статус **`3` (Отчислен)**. 
*   **Правильный приказ:** Для такого случая используется `orderTypeId = 3` (Отчисление) с категорией **`categoryId = 1008`** (*«В связи с окончанием срока обучения докторантуры»* / *«Due to end of PhD training period»*).
*   **Обоснование:** Без защиты диссертации невозможно сгенерировать номер диплома, а без номера диплома сущность `GRADUATES` не пройдет валидацию.
Если вуз ошибочно оформил приказ о выпуске (`orderTypeId=28`), а через несколько лет докторант приходит защищаться, **нельзя делать второй приказ о выпуске**.
*Правильный алгоритм:*
1. Отменить/откатить ошибочный статус выпускника (вернуть статус `3`).
2. Оформить приказ о **Восстановлении** (`orderTypeId=7`, `categoryId=172` для докторантуры) и перевести в **`status=1` (Обучается)** на время прохождения защиты. Обязательно выставить **последний курс** (`3` курс).
3. **Генерация номера диплома:** Согласно официальной инструкции, в веб-модуль на генерацию попадают обучающиеся **ТОЛЬКО со статусом `1` (Обучается)**, обучающиеся на последнем курсе, с освоенным порогом кредитов (для докторантуры ≥ 180 ECTS). В вузе генерируют номер и серию диплома.
4. **Приказ о выпуске:** После защиты и генерации номера вуз оформляет подлинный приказ о завершении обучения (`orderTypeId=28`) и переводит в `status=4`. Только тогда QR-код диплома активируется для проверки на сайте ЕПВО.
5. **Валидация GRADUATES:** При передаче записи докторанта в `GRADUATES` поле **`finishDocDate`** (Дата выдачи диплома) является **обязательным** (как и для всех степеней, кроме «Интерн» и «Резидент»). Отсутствие этой даты при наличии диплома приводит к HTTP 400.

#### Внешние слушатели (защита из другого вуза)
Докторанты, обучавшиеся в другом ОВПО, но защищающие диссертацию в вашем диссертационном совете, **не зачисляются** как обычные студенты (`STUDENT`). Для них внедрен процесс:
1. Запись создается в сущности **`EXTERNAL_LISTENER`** (Внешние слушатели).
2. Обязательно указывается **`mainUniversityId`** — ID того вуза, где они фактически учились.
3. Поле `type` в `EXTERNAL_LISTENER` устанавливается в соответствующий тип слушателя докторантуры. **Статус должен быть `1` (Слушатель), чтобы появиться в генерации.**
4. После успешной генерации диплома и защиты статус (`status`) в таблице `EXTERNAL_LISTENER` меняется на `4` (Выпущен). Головной вуз при этом в своей финальной таблице `GRADUATES` обязан отправить флаги: `defenseOfDissertationInOtherHpeo = true` и `defenseOfDissertationOtherHpeoId = <ID_вашего_вуза>`.
5. **Процедура выпуска:** Для перевода слушателя в статус «Выпускник» (`status=4`) необходимо:
    - Сгенерировать номер диплома в веб-интерфейсе (см. ниже).
    - Отправить `UPSERT` запрос в `EXTERNAL_LISTENER`, заполнив поля `status: 4`, `diplomaNumber`, `diplomaSeries`, `graduationYear` и `educationDocumentIssueDate`.
6. **Отсутствие приказов и транскриптов:** Для внешних слушателей (`EXTERNAL_LISTENER`) стандартная цепочка приказов в таблице `ORDERS` (зачисление/выпуск) **не используется**, так как сущность `ORDERS` требует `studentId`, которого нет у слушателя. Транскрипт также не требуется. Все переходы состояний осуществляются через прямое обновление поля `status`.
7. **Где посмотреть слушателя?** Слушатели **скрыты** из основного поиска «Контингент». Увидеть их можно только в модуле **«Генерация номеров документов»** (вкладка «Слушатели») или в отчётах по финансированию (для подготовительного отделения).
8. **API для проверки:**
    - Выгрузка списка: `GET /org-data/EXTERNAL_LISTENER/find-all-pageable`
    - Поиск по ID: `GET /org-data/EXTERNAL_LISTENER/find-by-id?id={"type":3,"listenerId":...}`
    - *Поиск по ИИН на сервере не поддерживается (фильтрация только на стороне клиента после выгрузки списка).*

### 2.15. Редактирование приказов (Замена студента)
Поскольку в ЕПВО архитектура приказов транзакционная, **заменить студента в уже проведенном приказе путем прямого редактирования ID нельзя** (композитные ключи). Шапка приказа (`ORDERS`) и параграф (`ORDER_SECTIONS`) остаются неизменными, а исправление выполняется перепривязкой:
1. Выполняется **DELETE**-запрос для записей `SECTION_PERSON` и `ORDER_STUDENT_INFO` ошибочного студента.
2. Отправляется запрос в сущность `STUDENT` ошибочного обучающегося для **отката его статуса** в прежнее состояние (например, если приказ изменил его с 1 на 4, вернуть на 1).
3. Выполняется **POST**-запрос новых записей `SECTION_PERSON` и `ORDER_STUDENT_INFO` для правильного студента к тому же параграфу (`sectionId`). Обязательно передается корректный `movementDate`.
4. Обновляется статус правильного студента в карточке `STUDENT` согласно логике приказа.

### 2.16. Модуль «Генерация номеров документов об образовании» — бизнес-правила

Источник: официальная инструкция «Инструкция ИС ЕПВО Генерация номеров документов ОВПО общая 09 02», Астана, 2024.

#### Предусловия появления обучающегося в списке на генерацию

| # | Условие | Поле / Проверка |
|---|---------|----|
| 1 | Статус **«Обучающийся»** | `status = 1` |
| 2 | **Не** в академическом отпуске | `isInRetire = 0` |
| 3 | Заполнен **ИИН** | `iin IS NOT NULL` |
| 4 | Текущий курс = **последний курс** по форме обучения | `courseNumber = courseCount` |
| 5 | ОП из Реестра ОП с правом выдачи **диплома собственного образца** | Привязка к направлению подготовки |
| 6 | Серия/номер **ранее не генерировались** | Однократная генерация |
| 7 | Наличие **программы финансирования** | Поле `fundingProgramId` должно быть заполнено (таблица `FUNDING_PROGRAM` / `center_funding_programs`), если студент обучается по гранту. Без него генерация не происходит. |

> ⚠️ **Критично:** Если студент уже переведён в `status=4` (Выпускник), он **исчезает** из вкладки «Генерация номеров» и номер диплома сгенерировать будет невозможно.

#### Предусловия для слушателей (`EXTERNAL_LISTENER`)

| # | Условие | Поле / Проверка |
|---|---------|----|
| 1 | Статус **«Слушатель»** | `status = 1` |
| 2 | Заполнен **ИИН** | `iin IS NOT NULL` |
| 3 | Серия/номер **ранее не генерировались** | Однократная генерация |

> Для слушателей проверка достаточности освоенных кредитов **не применяется**.

#### Пороговые значения кредитов (ECTS)

| Академическая степень | Срок обучения | Порог кредитов |
|---|:---:|:---:|
| Бакалавриат | 4 | 240 |
| Бакалавриат | 5 | 300 |
| Магистратура (научно-пед.) | 2 | 120 |
| Магистратура (профильная) | 1 | 60 |
| Магистратура (профильная) MBA | 1 / 2 | 60 / 120 |
| Докторантура (PhD) | 3 | 180 |
| Докторантура (по направлениям) | 3 | 180 |
| Докторантура (DBA) | 3 | 180 |

Пороговые значения определяются и могут быть изменены представителем МНВО РК.

#### Правила подсчёта освоенных кредитов (из транскрипта)
Дисциплина считается освоенной, если **все** условия выполнены:
1. Итоговая оценка **≥ 50** баллов (или «Зачёт» для формы контроля «Зачёт/не зачёт»).
2. Дисциплина **не удалена** из транскрипта.
3. Дисциплина **не была пройдена повторно** (пересдача не считается).
4. Дисциплина **не является дополнительно изученной**.
5. Учитываются все виды учебной нагрузки: дисциплины, практики, НИР, ИГА.

#### Процедура генерации
1. Оператор ОВПО отмечает обучающихся/слушателей → нажимает «Сгенерировать».
2. Система запрашивает подтверждение → открывает окно выбора сертификата **ЭЦП** (ключ RSA, приложение NCALayer).
3. Проверяется: ИИН и ФИО пользователя = ЭЦП, БИН организации = ЭЦП, сертификат подтверждён НУЦ РК.
4. Присваиваются серии и генерируются номера → данные сохраняются во вкладке «История».
5. **QR-код** для проверки подлинности (`https://epvo.kz/#/verification_education_document/...`) активируется **только после** `status=4` (Выпущен).

### 2.17. Транскрипты и пересдачи (FX)

В ЕПВО транскрипт должен отражать реальную академическую историю студента.

**Как обрабатываются пересдачи:**
1. **Методология ЕПВО:** ОВПО **должны** передавать все попытки студента, включая двойки (F/FX) и последующие пересдачи (например, записи из летнего семестра). 
2. **Задвоение кредитов:** Передача пересдач не приводит к двойному начислению кредитов. На стороне ЕПВО работает специальный алгоритм фильтрации (см. §2.16, правило №3: *"Дисциплина, которая не была пройдена повторно"*). Система самостоятельно находит две одинаковые дисциплины в транскрипте, отбрасывает проваленную попытку и плюсует кредиты только за успешную оценку.
3. **Технический нюанс:** Если интегрирующая система (например, Platonus) технически отправит только *финальные положительные оценки*, скрыв историю пересдач, API ЕПВО вернёт HTTP 200 OK. Модуль генерации так же штатно сложит кредиты. Однако такая практика нарушает Data Governance, так как цифровой транскрипт становится неполным.

> **Вывод:** Рекомендуется передавать транскрипт в исходном виде, со всеми летними семестрами и пересдачами. ЕПВО корректно обработает их при подсчете ECTS.

### 2.18. Инвалидность и особые потребности

Учет контингента с инвалидностью и ОВЗ (ограниченными возможностями здоровья) влияет на три ключевых модуля: **Профиль студента**, **Стипендии** и **Инфраструктура ОВПО**.

**1. Профиль обучающегося (`STUDENT` / `STUDENT_INFO`)**
- В сущности `STUDENT` передается поле `disabilityCategoryId` (тип `int32`). 
- Значение берется из центрального системного справочника ЕПВО `DisabilityCategories` («Категории лиц с инвалидностью»). Дополнительно существует служебный справочник `special_needs_disability_categories` (связь «категория инвалидности — условие/потребность»).

**2. Финансирование и Стипендии (`SCHOLARSHIP`)**
- Существуют отдельные типы стипендий (`scholarshipTypeId`):
  - *Инвалиды по зрению/слуху*
  - *Инвалиды/Участники ВОВ*
  - *Академ. отпуск по мед. показаниям (инвалиды)*
- **⚠️ Критическое правило отчетов:** При отправке ежемесячного отчета в Финсцентр (к 15-му числу), сканы подтверждающих документов (справки ВТЭК, мед. заключения) для обучающихся с инвалидностью и сирот **необходимо прикреплять каждый месяц**.

**3. Инфраструктура и Общежития (`DORMITORY` / ВП-7)**
В рамках отчетов СУР и Национального рейтинга ОВПО строго мониторится доступность университетской инфраструктуры (зданий, общежитий) для контингента с особыми потребностями.
В сущности зданий, а также выпускников и общежитий предусмотрены счетчики:
- `specialToiletsCount` — количество оборудованных санузлов для лиц с инвалидностью.
- `rampCount` — наличие/количество пандусов.
- `liftCount` — количество специализированных лифтов.

### 2.19. Государственные экзамены и ИГА (ГАК/ГЭК)

Передача результатов итоговой государственной аттестации (ИГА) и государственных экзаменов имеет распределенную архитектуру. Основной массив данных отправляется не в общий транскрипт, а в специализированную таблицу **`GENERAL_EXAMS_MARKS`**. 

**1. Сущность `GENERAL_EXAMS_MARKS` (Оценки по гос. аттестации)**
Служит для сбора результатов заседаний ГАК и ГЭК. В ней передаются:
- `examId` — ID типа аттестации (комплексный экзамен, защита дипломной работы/диссертации). Связь со справочником `Q_EXAMINATIONS`.
- `name` — текстовое название экзамена.
- `reportId` — идентификатор ведомости (ссылка на протокол).
- **`sacNumber` / `sacDate`** — номер и дата утверждения протокола ГАК/ГЭК.

**2. Сущность `Q_EXAMINATIONS` (Справочник видов ИГА)**
Перед загрузкой оценок, ОВПО должно выгрузить свои типы гос. экзаменов в этот справочник. Форма завершения дисциплин/ИГА определяется вузом.

**3. Дублирование в `TRANSCRIPT` (Транскрипт)**
Результаты ИГА частично отражаются в транскрипте для корректного подсчета финальных ECTS кредитов (в модуле "Генерация дипломов" учитываются дисциплины, практики, НИР, ИГА):
- Передается флаг `isGeneralExam = true`.
- Используется поле `hash` (ID оценки за гос. экзамен) для связи с `GENERAL_EXAMS_MARKS`.
- **Исключение:** Дисциплина "История Казахстана", если она сдается как госэкзамен, должна отправляться **и в `TRANSCRIPT`** (для кредитов), **и в `GENERAL_EXAMS_MARKS`** (для фиксации факта сдачи ГЭК).

**4. Сущность `GRADUATES` (Выпускники)**
При конечном выпуске статус прохождения и документальная основа снова дублируется:
- `iacProtocol` — номер протокола ГАК/ГЭК.

> **Вывод:** Для корректной интеграции ИГА требуется синхронизированная отправка трех сущностей в строгом порядке: `Q_EXAMINATIONS` → `GENERAL_EXAMS_MARKS` → `TRANSCRIPT` (с флагами ИГА), после чего данные идут в `GRADUATES`.

### 2.20. Архитектура финансирования (Форма оплаты и Программы)

Для корректной работы модуля "Финансирование" и начислений со стороны АО "Финансовый центр" критически важно своевременно заполнять три связанных, но семантически разных атрибута:

**1. Форма оплаты (`paymentFormId`)**
- Технически: обновляется в карточке `STUDENT` и `GRADUATES` в виде ID из справочника `PaymentForms`.
- **Справочник значений:**
  - `paymentFormId = 1` — Платная (обучение за счет собственных средств).
  - `paymentFormId = 2` — Бесплатная (Государственный образовательный грант). *Именно это слово отображается в цифровых справках eGov/Kaspi. Все целевые, ректорские гранты и гранты МИО агрегируются ЕПВО в категорию "Бесплатная".*
- Заполнение: Первично при зачислении. Вторично — при издании **приказа о смене формы оплаты** (код `10`). Если студент выигрывает вакантный грант, необходимо отправить приказ, а следом немедленно обновить (`UPSERT`) карточку `STUDENT` с новым значением `paymentFormId = 2`.

**2. Вид программы финансирования (`fundingProgramId` / `FUNDING_PROGRAM`)**
- Технически: ID из справочника `center_funding_programs` (например, 204 «Обеспечение кадрами с высшим образованием»). Отправляется в составе профиля `STUDENT_INFO` или отдельной сущностью `FUNDING_PROGRAM`.
- Заполнение: Одновременно с зачислением или получением государственного гранта. Без этого поля студент с грантом будет отображаться с пометкой "НЕТ" в карточке Финансирования и не попадёт в стипендиальную ведомость.

**3. Вид/Источник финансирования (Source of finance)**
- Технически: атрибут (`moneySource`, или ID из справочника `source_of_finance`) используется в узких контекстах, таких как Академическая мобильность (текстовое поле источника), квалификация ППС (`paymentFormId` = средства вуза/РБ) или для показателей Стратегического управления (СУР), где учитываются НИР и целевые субсидии.

### 2.21. Реестр образовательных программ (TFW-9, TFW-10)

Система ЕПВО различает Группы образовательных программ (ГОП) и сами ОП.
*   **Два типа ГОП**: В одной таблице `PROFESSION` лежат и старые специальности (`classifier=1`) и новые ГОП (`classifier=2`).
*   **Иерархия (JOIN-путь)**: ОП (`SPECIALIZATIONS`) не ссылается напрямую на ГОП. Она привязана к мостовой таблице `PROFESSION_CAFEDRA` через `prof_caf_id`. 
    Логическая связь: `SPECIALIZATIONS.prof_caf_id → PROFESSION_CAFEDRA.id → PROFESSION.professionid`.
*   **Статусы Реестра**: Отправка по API `statusep=1` НЕ включает физически ОП в реестр. Требуется официальная заявка через [enic-kazakhstan.edu.kz](https://enic-kazakhstan.edu.kz/ru/reestr-op/ovpo-1).
*   **Скрытие из расчетов (СУР)**: Управление параметрами устойчивого развития (рейтингами) осуществляется флагом `ignore_rms`.
*   **Правило Full Replace**: При обновлении `PROFESSION` или `SPECIALIZATIONS` через `/org-data/list/save` необходимо передавать ВСЕ поля объекта, иначе пропущенные поля будут сброшены в `null`.

### 2.22. Работа с архивами и старыми выпускниками (Исторические данные)

При синхронизации данных о выпускниках, окончивших обучение по старым программам, действуют особые «ослабленные» правила валидации:

1.  **Академическая степень**: Используется **`degreeId = 8`** (Специалитет).
2.  **Автономность записи**: В отличие от новых наборов, старые выпускники **не требуют** наличия параллельной записи в таблице `STUDENTS` со статусом `3`. Запись в `GRADUATES` может существовать и отображаться в модуле выпускников автономно.
3.  **Валидация связей (Degree ↔ Form/GOP)**: Правило соответствия степени её форме обучения и ГОП в центральном справочнике для старых наборов **не является блокирующим**. Запись будет отображаться в отчётах даже при семантических расхождениях.
4.  **Валидация GPA**: Для зачисленных до 2010 года (`enrollYear < 2010`) поле `GPA` — **необязательно** (может быть `null`).
5.  **Валидация ИИН**: При ошибках в 7-м разряде ИИН (старые выдачи) использовать флаг `iin_gived_by_government = true` в `GRADUATES`.

### 2.23. Синхронизация Выпускников: Новые наборы (New Intakes)

Для студентов, чей жизненный цикл полностью отслеживается в ЕПВО, действуют максимально строгие правила целостности:

1.  **Правило двойной записи (Dual-Entity)**: Выпускник **обязан** существовать одновременно в двух сущностях:
    - `STUDENT` — со статусом `status=4` (или `isStudent=3`).
    - `GRADUATES` — полный профиль.
    Рассинхронизация данных между ними (разные ИИН, даты рождения) приведет к ошибке синхронизации.
2.  **Строгая валидация связей**: Если в `GRADUATES` указан `degreeId`, а `studyformid` или `professionid` (ГОП) к этой степени **не относятся** согласно центральному справочнику ЕПВО — запись **не пройдет валидацию** и будет скрыта из отчетов.
3.  **GPA**: Обязателен (`not null`) для всех зачисленных после 2010 года.

> 🎓 **Вывод:** Причины «пропадания» выпускника из отчетов зависят от года его поступления. Для новых наборов ищите несоответствия в связке «Степень-Форма-ГОП», для старых — проверьте статус автономной записи в `GRADUATES`.

### 2.25. Недокументированные поля STUDENT / STUDENT_INFO (расхождение OpenAPI ↔ Runtime)

Ряд полей отсутствует в OpenAPI-спецификации (`RF_TFW-1.3`), но физически существует в БД ЕПВО и требуется модулем «Финансирование». Эти поля были добавлены Platonus позже и задокументированы только в чате техподдержки и/или официальных инструкциях ОВПО.

#### 1. `fundingID` (таблица `students`)

| Атрибут | Значение |
|---------|----------|
| **Сущность** | `STUDENT` |
| **Тип** | `int` |
| **В Swagger** | «Вид программы финансирования» |
| **FK** | → `center_funding_programs` (центральный справочник `FUNDING_PROGRAM`) |

**Значения:**

| `fundingID` | Описание |
|:-----------:|----------|
| `1` | Подпрограмма 100 — «Подготовка специалистов с высшим, послевузовским образованием» |
| `2` | Подпрограмма 118 — «Стипендиальные программы для иностранных граждан / диаспора» |
| `-1` | Платная основа (только для студентов БЕЗ гос. гранта) |

> ⚠️ **Критично:** Без `fundingID = 1 или 2` студент-грантник **НЕ попадёт** в отчёт модуля «Финансирование. Стипендии». Источник: чат ЕПВО (2025-H1, первое объявление; 2025-H2, расширенная версия с добавлением значения `-1`).
